# Анализ ошибок подключения MS Access к SQL Server через RDS (Basis Workplace)

**Дата:** 2026-06-02  
**Среда:** MS Access → ODBC DSN `FishEye` → SQL Server `SPB-05-NV-SQL1` через RDS-хост `NV-SK-TSW112.adm.gazprom.ru`  
**Клиент удалённого доступа:** Basis Workplace  
**Проявление:** периодические ошибки `ODBC – ошибка подключения к 'FishEye'`, `Named Pipes Provider: Could not open a connection to SQL Server [5]`, `TCP Provider: Требуемый адрес для своего контекста неверен`, `Automation error -2147417848`

**Скриншоты ошибок:** не в Git; локально на nb-win — каталог и пути в [README.md](README.md) (`\\wsl.localhost\Ubuntu\home\alex\projects\femsq\docs\sql_err\images\`).

## 1. Цепочка уровней доступа

Уровни перечислены от общего к частному. Каждый уровень может быть источником ошибки или точкой её устранения.

---

### Уровень 1 — Basis Workplace (брокер/портал)

**Что это:**  
Платформа управления виртуальными рабочими столами и терминальными сеансами. Принимает запрос на подключение пользователя и решает: вернуть его в **существующий отключённый сеанс** или создать **новый**. Управляет пулом RDS-хостов и ВМ, таймаутами завершения сеансов.

**Влияние на ошибки:**  
- Если Basis возвращает пользователя в тот же Session ID — **ошибки сохраняются**, потому что сетевой стек сессии не обновляется.  
- Если Basis выдаёт новый сеанс или другой хост из пула — пользователь получает чистую среду и **ошибки временно исчезают**.  
- Настройки таймаутов в Basis определяют, как долго живёт «отключённый» сеанс и происходит ли его **автозавершение** (что выглядит как «само исчезло и заработало»).

**Признак этого уровня:**  
В интерфейсе клиента Basis доступна только кнопка **«Отключить»** без «Выхода»/«Log off» — это намеренное административное ограничение. Иногда файлы в «Документах» и DSN исчезают без явного действия пользователя — Basis завершил сеанс по таймауту.

**Действия для устранения ошибки:**

| Действие | Подробности | Кто выполняет |
|----------|-------------|---------------|
| Выйти через клиент Basis | Найти пункт «Выход» / «Log off» / «Завершить сеанс» (не «Отключить») в меню клиента, если он доступен | Пользователь |
| Запросить сброс сеанса у администратора | Администратор Basis или сервера видит ваш сеанс в консоли и может принудительно завершить его (log off) | Администратор |
| Закрепить постоянный хост | Попросить администратора настроить «персональный» пул или закрепить пользователя за конкретным `NV-SK-TSW112`, чтобы профиль и файлы не терялись | Администратор |
| Настроить таймаут автозавершения | В настройках Basis/RDS GPO: «End a disconnected session» = например, 30 минут вместо «Never», чтобы накопленный больной сеанс завершался сам | Администратор |

### Уровень 2 — RDS-хост (`NV-SK-TSW112.adm.gazprom.ru`)

**Что это:**  
Физический или виртуальный сервер Windows Server (идентифицирован в Event Viewer как `NV-SK-TSW112.adm.gazprom.ru`), на котором одновременно работают 33 и более пользовательских сеансов. Все сеансы **разделяют** общие ресурсы сервера: TCP-порты, память, процессорное время.

**Влияние на ошибки:**  
Главный источник **Event ID 4227** (источник `Tcpip`). Когда суммарная частота открытия/закрытия TCP-соединений от всех пользователей высока, эфемерные порты заканчиваются на **уровне всего хоста**. При этом **ни один** пользователь в этот момент не может установить новое TCP-соединение — ни к SQL Server, ни по `telnet`. В журнале зафиксированы события 4227 в **09:34, 11:47, 13:06, 15:50** (01.06.2026) и систематически в **22:00** многих дней (вероятно, фоновый batch-процесс).  
События **1064 от TerminalServices-RemoteConn** в те же моменты говорят о том, что исчерпание портов рвёт и сами RDS-сессии.

**Признак этого уровня:**  
- `telnet SPB-05-NV-SQL1 1433` **не работает** в момент ошибки (запущен внутри сессии).  
- ODBC-тест DSN `FishEye` даёт `TESTS FAILED` с TCP-ошибкой.  
- Проблема возникает не у одного, а у **нескольких** пользователей одновременно.

**Действия для устранения ошибки:**

| Действие | Подробности | Кто выполняет |
|----------|-------------|---------------|
| Уменьшить `TcpTimedWaitDelay` | Реестр: `HKLM\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters`, параметр `TcpTimedWaitDelay` (DWORD), значение `30` (секунды вместо 240 по умолчанию). Порты освобождаются в 8 раз быстрее. Требует перезагрузки. | Администратор |
| Расширить диапазон эфемерных портов | `netsh int ipv4 set dynamicportrange tcp start=10000 num=55535` — увеличивает доступный пул с ~16 000 до ~55 000 портов. Не требует перезагрузки. | Администратор |
| Включить ODBC Connection Pooling | 32-bit ODBC Administrator (`SysWOW64\odbcad32.exe`) → вкладка **«Пул соединений»** → `SQL Server Native Client 11.0` → двойной клик → `CPTimeout = 60`. Соединения переиспользуются, не создаются каждый раз заново. | Администратор |
| Найти ночной процесс (22:00) | В Event Viewer на сервере в ~22:00 — какой процесс создаёт лавину соединений. Вероятно, резервное копирование или scheduled task. Устранить или перенести по времени. | Администратор |

### Уровень 3 — Сеанс Windows (Session ID)

**Что это:**  
Изолированный контекст пользователя на RDS-хосте. Имеет уникальный **Session ID** (например, 33 — видно из пути `AppData\Local\Temp\33\...` в логах ODBC). Содержит все запущенные процессы пользователя (Access, Excel, ODBC-драйверы), переменные среды и собственное состояние Winsock.

**Влияние на ошибки:**  
При высокой нагрузке (уровень 2) Winsock-контекст сессии «деградирует»: попытка создать любой исходящий TCP-сокет из этой сессии даёт `WSAEADDRNOTAVAIL`. Именно поэтому в момент ошибки **и ODBC, и telnet** из той же сессии не работают — это не проблема SQL Server, а проблема **сессии на RDS-хосте**.  
Кнопка «Отключить» в Basis/Windows **сохраняет** Session ID. При повторном подключении Basis возвращает пользователя в **тот же** сеанс — деградированное состояние сохраняется, ошибки не исчезают.

**Признак этого уровня:**  
Выполнить в cmd в удалённой сессии:
```cmd
query user
```
Если Session ID **одинаковый** до и после «переподключения» — вы в той же сессии. Файлы в «Документах» и DSN **остаются** — подтверждение.

**Действия для устранения ошибки:**

| Действие | Подробности | Кто выполняет |
|----------|-------------|---------------|
| Полный выход из сеанса | `Ctrl+Alt+End` в окне удалённого доступа → **«Выйти»** (не «Отключить»). Это завершает Session ID, при следующем входе создаётся новый сеанс. | Пользователь |
| `logoff` в командной строке | Открыть cmd в удалённой сессии, выполнить `logoff`. Сеанс завершится — соединение с удалённым рабочим столом оборвётся. Войти заново. | Пользователь |
| `shutdown /l` | Альтернатива `logoff`, тот же эффект. | Пользователь |
| Принудительный сброс администратором | На сервере: `query user` → найти Session ID → `logoff <id>`. Или через консоль Basis. Нужно, если в меню пользователя нет «Выхода». | Администратор |

### Уровень 4 — Сетевой стек сессии (Winsock / TCP-порты)

**Что это:**  
Механизм TCP/IP Windows, через который все процессы сессии создают сетевые соединения. Эфемерные порты (диапазон ~49152–65535) выдаются каждому новому исходящему TCP-соединению. После закрытия соединения порт остаётся в состоянии **TIME_WAIT** около **240 секунд** (4 минуты), прежде чем стать доступным снова. При высокой частоте открытия/закрытия соединений порты заканчиваются.

**Влияние на ошибки:**  
Непосредственная техническая причина **всех** наблюдаемых ошибок:
- `Named Pipes Provider: Could not open a connection [5]` — попытка Named Pipes при деградированном стеке.
- `TCP Provider: Требуемый адрес для своего контекста неверен` (`WSAEADDRNOTAVAIL`, код 10049) — нет свободного локального порта для привязки TCP-сокета.
- `Login timeout expired` — следствие: до сервера не дошли.

Winsock-каталог сессии проверен командой `netsh winsock show catalog` — он **чистый**, сторонних LSP (от антивируса или VPN) нет. Проблема не в повреждении Winsock, а в **исчерпании ресурсов**.

**Признак этого уровня:**  
```cmd
netstat -no -p tcp | findstr ":1433" | findstr "TIME_WAIT"
netsh int ipv4 show dynamicportrange tcp
```
Много строк TIME_WAIT к :1433 — подтверждение.

**Действия для устранения ошибки:**

| Действие | Подробности | Кто выполняет |
|----------|-------------|---------------|
| Подождать ~5–10 минут | TIME_WAIT истекает сам. Работает только если нагрузка уже прекратилась. Ненадёжно при активных пользователях. | Пользователь |
| Завершить сеанс (уровень 3) | Новый сеанс — чистый сетевой контекст. | Пользователь |
| Уменьшить `TcpTimedWaitDelay` (на сервере) | Уже описано на уровне 2. | Администратор |

### Уровень 5 — Профиль пользователя Windows

**Что это:**  
Папка `C:\Users\<логин>\` на RDS-хосте: файлы в «Документах», куст реестра `HKCU` (содержит **User DSN** `FishEye` и пользовательские настройки). При каждом входе Windows загружает этот профиль.

**Влияние на ошибки:**  
- **User DSN** `FishEye` хранится в `HKCU\Software\ODBC\ODBC.INI\FishEye` и в `HKCU\Software\WOW6432Node\ODBC\ODBC.INI\FishEye` (32-bit ветка). При смене хоста в пуле Basis или создании нового профиля — DSN **исчезает**, Access не может подключиться к FishEye вообще (другая ошибка — DSN не найден).  
- Файлы `.accdb` в «Документах» — при смене хоста могут остаться на **предыдущем** сервере.

**Признак этого уровня:**  
После переподключения **файлы и/или DSN исчезли** — вы на другом хосте или с новым профилем.

**Действия для устранения ошибки:**

| Действие | Подробности | Кто выполняет |
|----------|-------------|---------------|
| Пересоздать User DSN `FishEye` | `C:\Windows\SysWOW64\odbcad32.exe` → вкладка «Пользовательский DSN» → «Добавить» → `SQL Server Native Client 11.0` → имя `FishEye`, сервер `SPB-05-NV-SQL1`, Windows auth, БД `FishEye`. Занимает 2–3 минуты. | Пользователь |
| Хранить `.accdb` на сетевом диске | Переместить рабочие файлы Access на `X:\grp\...` или иной сетевой ресурс, доступный с любого хоста. Тогда смена хоста не приводит к потере файлов. | Пользователь |
| Системный DSN | Если администратор создаст DSN в разделе «Системный» (а не «Пользовательский»), он будет доступен всем пользователям на данном хосте и не будет привязан к конкретному профилю. | Администратор |

### Уровень 6 — ODBC DSN `FishEye` и драйвер

**Что это:**  
32-разрядный User DSN с именем `FishEye`: драйвер `SQL Server Native Client 11.00.7462` (SNAC 11, выпуск 2012 года), сервер `SPB-05-NV-SQL1`, Windows Integrated Security, база данных `FishEye`. Используется MS Access как источник данных для связанных таблиц и pass-through запросов.

**Влияние на ошибки:**  
- DSN настроен **без явного протокола**, поэтому SNAC при каждом новом соединении сначала пробует **Named Pipes**, затем **TCP**. Обе попытки падают при деградации сетевого стека.  
- ODBC Connection Pooling **выключен** → каждый `QueryDef` или `OpenRecordset` к серверу инициирует **новый** TCP-сокет, что увеличивает churn и ускоряет исчерпание портов.  
- Старый драйвер SNAC 11 хуже обрабатывает сетевые ошибки, чем современные.

**Признак этого уровня:**  
В диалоге «SQL Server Login» Access показывает `Data Source: FishEye` и `Use Trusted Connection: Yes`. Тест DSN (`SysWOW64\odbcad32.exe` → «Проверить») показывает `TESTS FAILED` с конкретной TCP/Named Pipes ошибкой.

**Действия для устранения ошибки:**

| Действие | Подробности | Кто выполняет |
|----------|-------------|---------------|
| Указать явный протокол TCP в DSN | Пересоздать DSN, в поле «Сервер» ввести `tcp:SPB-05-NV-SQL1,1433` — исключает попытку Named Pipes. Результат: при исправном стеке подключается быстрее; при деградированном — всё равно упадёт, но с более понятной ошибкой. | Пользователь |
| Включить ODBC pooling | На уровне сервера: 32-bit ODBC Administrator → вкладка «Пул соединений» → `SQL Server Native Client 11.0` → `CPTimeout = 60`. Снижает число новых TCP-соединений. | Администратор |
| Обновить драйвер на ODBC Driver 18 | Скачать **Microsoft ODBC Driver 18 for SQL Server (x86)** с сайта Microsoft, установить. Пересоздать DSN `FishEye` с новым драйвером (имя DSN оставить `FishEye` — Access не заметит замены). Новый драйвер лучше работает на RDS и имеет меньше известных проблем с TIME_WAIT. | Пользователь / Администратор |

### Уровень 7 — VBA-код MS Access

**Что это:**  
Модули и формы `.accdb`: `dbAccess.cls`, `ra_aAgFee23_06.cls`, `Form_ra_a.cls` (процедура `btnAuditRun_Click`), `Form_ipgChMin.cls` (процедура `btnMasteringPercent_2408_Click`). Они создают ODBC-соединения (через DAO QueryDef с `.Connect` или через ADODB) при каждом обращении к SQL Server.

**Влияние на ошибки:**  
- В `ra_aAgFee23_06.Audit` каждый раз создаётся **~16 новых `QueryDef`** с `ODBC;DSN=FishEye;...` на один лист — при R строках актов (сотни–тысячи) + R построчных вставок в linked table → высокая частота open/close соединений → ускоряет достижение Event 4227.  
- В `dbAccess.GetNewADODBConnection()` при каждом вызове создаётся **новый** `ADODB.Connection` (а значит, новый TCP-сокет). Кэширования нет.  
- В `SqlSrvRecordsetBySqlString()` `QueryDef` не закрывается явно — ресурс удерживается до сборщика мусора VBA.  
- При ошибке Access показывает пользователю техническое сообщение и прерывает работу без возможности повтора.

**Признак этого уровня:**  
Ошибка появляется **в середине** длинной операции (лог «ход ревизии» показывает несколько успешных записей, потом `ODBC – ошибка FishEye`). Если бы проблема была «всегда» — ошибка была бы с первой же операции.

**Действия для устранения ошибки:**  
Подробные фрагменты кода «было → стало» — в файле `vba-fix-proposal.md` (рядом с этим документом).

| Изменение | Модуль / процедура | Эффект |
|-----------|--------------------|--------|
| Добавить `Private moCn As ADODB.Connection` | `dbAccess.cls`, секция переменных | Инфраструктура для кэширования соединения |
| Кэшировать ADODB: возвращать открытое соединение, не создавать новое | `dbAccess.cls`, `GetNewADODBConnection()` | 1 TCP-сокет на весь объект вместо N |
| Добавить `Class_Terminate` с закрытием `moCn` | `dbAccess.cls` | Гарантированное освобождение порта |
| Retry 3 × 3 сек при ошибке подключения | `dbAccess.cls`, `GetNewADODBConnection()` | Переживает кратковременный port exhaustion; внятное сообщение пользователю |
| `qd.Close: Set qd = Nothing` после `OpenRecordset` | `dbAccess.cls`, `SqlSrvRecordsetBySqlString()` | Немедленное освобождение ODBC-ресурса |
| Убрать `objConn.Close` | `Form_ipgChMin.cls`, `btnMasteringPercent_2408_Click()` | Не закрывает кэшированное соединение раньше `Class_Terminate` |
| Один переиспользуемый `QueryDef` на всю `Audit` | `ra_aAgFee23_06.cls` | Снижает ~16 open/close до 1 на лист (отдельная доработка) |
| Пакетная загрузка вместо R строк `AddNew` | `ra_aAgFee23_06.cls`, `AuditFillFromSource` | Снижает R round-trip до 1 вызова хранимки (отдельная доработка) |

### Уровень 8 — TCP-соединение к SQL Server (`SPB-05-NV-SQL1:1433`)

**Что это:**  
Конкретный TCP-сокет между RDS-хостом и SQL Server 2012 SP4. Сам SQL Server работает штатно — проблема не в нём.

**Влияние на ошибки:**  
Конечная точка отказа. Если все уровни выше исправны — соединение устанавливается за миллисекунды. Падает только как **следствие** проблем на уровнях 3–4. Менять ничего на этом уровне не нужно.

## 2. Итоговая логика возникновения ошибки

```
[Basis направил в тот же отключённый сеанс]         ← уровень 1
   ↓
[Сеанс жив (тот же Session ID), Winsock деградировал]
   ← причина: много пользователей + часто открываемые соединения
   ← Event 4227 на хосте NV-SK-TSW112                ← уровень 2
   ↓
[Новый TCP-сокет из сессии невозможен]
   ← WSAEADDRNOTAVAIL / Error 5                       ← уровень 4
   ↓
[ODBC DSN FishEye пробует Named Pipes → TCP → оба падают]
   ← нет явного протокола, нет pooling                ← уровень 6
   ↓
[VBA-код получает ODBC Error 3151 / SQLState 08001]
   ← Access показывает диалог «SQL Server Login»
     или «ODBC – ошибка подключения к FishEye»        ← уровень 7
   ↓
[TCP-соединение к SPB-05-NV-SQL1:1433 не установлено] ← уровень 8
```

---

## 3. Цепочка «лечения» по приоритету

### 3.1 Прямо сейчас (без ожидания администратора)

**Шаг 1 — Попробовать полный выход из сеанса:**

```
Ctrl+Alt+End → нажать «Выйти» / «Sign out»
```

Если пункта «Выйти» нет — открыть командную строку (`Win + R → cmd`) и выполнить:

```cmd
logoff
```

Соединение с удалённым рабочим столом оборвётся — это ожидаемо. Войти снова через Basis.

**Шаг 2 — Убедиться, что это новый сеанс:**

После входа в командной строке выполнить:

```cmd
query user
```

Если Session ID **изменился** по сравнению с тем, что был до выхода — вы в новом сеансе. Проверить: запустить `telnet SPB-05-NV-SQL1 1433` — должно подключиться. Затем проверить работу Access.

**Шаг 3 — Если log off недоступен: подождать 5–15 минут**

Иногда помогает при лёгком port exhaustion: TIME_WAIT портов истекает, следующее соединение проходит. Ненадёжно при большой нагрузке.

---

### 3.2 Перед каждой длинной операцией (профилактика)

1. **Не запускать ревизию несколько раз подряд** без пауз — каждый прогон создаёт сотни SQL-обращений.  
2. Перед длинной ревизией — убедиться, что `telnet SPB-05-NV-SQL1 1433` **отвечает** (быстрая проверка сети).  
3. По возможности — запускать ревизию в **первой половине дня** (меньше накопленной нагрузки).  
4. **Закрывать Access и Excel** перед отключением от Basis (не оставлять открытыми в фоне).

### 3.3 Что сохранить, чтобы не потерять при «исчезновении»

Поскольку иногда после переподключения профиль меняется (другой хост пула, новый профиль):

| Что | Где хранить надёжно |
|-----|---------------------|
| Рабочий файл `.accdb` | Сетевой диск (`X:\grp\...`), не только `C:\Users\...\Documents\` |
| Параметры DSN `FishEye` | Записать: драйвер `SQL Server Native Client 11.0`, сервер `SPB-05-NV-SQL1`, БД `FishEye`, Windows auth |
| Шаблон пересоздания DSN | Инструкция ниже (раздел 3.4) |

### 3.4 Инструкция: пересоздание DSN FishEye за 2–3 минуты

Выполнять, если после входа DSN `FishEye` исчез.

1. Нажать **Win + R** → ввести:
   ```
   C:\Windows\SysWOW64\odbcad32.exe
   ```
   Нажать **OK**.

2. Открыть вкладку **«Пользовательский DSN»** → нажать **«Добавить…»**.

3. Выбрать драйвер **`SQL Server Native Client 11.0`** → **«Готово»**.

4. Заполнить поля:
   - **Имя:** `FishEye`
   - **Описание:** (оставить пустым)
   - **Сервер:** `SPB-05-NV-SQL1`  
     *(или `tcp:SPB-05-NV-SQL1,1433` — явное указание TCP)*

5. Нажать **«Далее»** → выбрать **«Со встроенной проверкой подлинности Windows»** → **«Далее»**.

6. Включить **«Изменить базу данных по умолчанию на»** → выбрать **`FishEye`** → **«Далее»** → **«Готово»**.

7. Нажать **«Проверить источник данных…»**. Должно появиться сообщение **`TESTS COMPLETED SUCCESSFULLY!`**. Если тест не прошёл — сетевой стек сессии ещё не восстановился; повторить после нового входа.

8. Нажать **OK**.

### 3.5 Для администраторов (среднесрочно)

| Приоритет | Действие | Эффект |
|-----------|----------|--------|
| 1 | `TcpTimedWaitDelay = 30` в реестре на `NV-SK-TSW112` + перезагрузка | Порты освобождаются в 8 раз быстрее |
| 2 | `netsh int ipv4 set dynamicportrange tcp start=10000 num=55535` | Увеличить пул портов с ~16K до ~55K |
| 3 | ODBC pooling: `CPTimeout=60` для SNAC 11 в 32-bit ODBC Admin | Соединения переиспользуются — меньше open/close |
| 4 | Найти и перенести/устранить ночной batch-процесс (~22:00) | Устранить хроническую перегрузку |
| 5 | Настроить log off по таймауту в Basis/GPO (например, 30–60 мин для disconnected) | Старые больные сеансы завершаются автоматически |
| 6 | Закрепить пользователя за постоянным хостом в пуле Basis | Профиль, файлы и DSN не теряются |

Общие принципы:

## 4. Диагностические команды (в cmd внутри удалённой сессии)

Выполнять **в момент ошибки**, не после:

```cmd
:: Базовая проверка TCP к SQL Server
telnet SPB-05-NV-SQL1 1433

:: Проверить Session ID
query user

:: Соединения в состоянии TIME_WAIT к SQL Server
netstat -no -p tcp | findstr ":1433" | findstr "TIME_WAIT"

:: Общее количество TIME_WAIT на хосте
netstat -no -p tcp | find /c "TIME_WAIT"

:: Диапазон эфемерных портов (администратор)
netsh int ipv4 show dynamicportrange tcp

:: Winsock-каталог (проверка на сторонние LSP)
netsh winsock show catalog
```
