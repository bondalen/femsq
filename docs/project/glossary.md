## Г

### Группа планов (`ipgUtPlGr`)

Набор планов агентов (`ipgUtPlGrP`, `ipgUtPlP`), привязанный к инвестпрограмме через `ipgChRlV.ipgcrvUtPlGr`. Определяет, какие строки уточнённого плана участвуют в расчёте освоения для данной ИП в цепи.

**Не путать с:** ревизией факта, сменой актуальности ИП.

---

## И

### Инвестпрограмма (ИП)

Версия инвестиционной программы в рамках цепи (`ipgCh`). Имеет срок актуальности в цепи: `ipgcrvStr` … `ipgcrvEnd` в таблице `ipgChRlV`. На конкретную дату расчёта актуальна **не более одной** ИП цепи.

**Не путать с:** ревизией (синхронизация факта Excel ↔ SQL).

**См. также:** смена актуальности инвестпрограммы, цепь инвестпрограмм, пункт инвестпрограммы.

---

## Н

### Накопленный план

Сумма помесячных плановых показателей UtPl с начала расчётного года до контрольной даты (`dAll` / `@MounthEndDate`). В стеке `_2606` отображается в полях `smmTtl`, `agSmmTtl` и аналогах PercentBrn (`ag_PlAccum`).

На дате окончания периода ИП (`ipgcrvEnd`) накопленный план равен cum UtPl на эту дату и **не обязан** совпадать с годовым лимитом, если ИП снята раньше 31.12.

---

## О

### Освоение лимита

Показатель исполнения инвестиционного лимита: соотношение факта (принято, представлено и др. категории) и плана на контрольную дату. Рассчитывается в контуре **расчёта освоения** (`ipgChMin`, стек `_2606`), а не в контуре ревизии факта.

---

## П

### Плановые показатели уточнённого плана (UtPl)

Помесячные суммы по пункту инвестпрограммы и элементам структуры затрат. Источник **плана** в mastering, `fnStCostRsIpgPn_2606` и PercentBrn стека `_2606`.

**Каноническое хранилище помесячной разбивки по stCost:** таблица `ipgUtPlPnLmMn` (см. отдельную статью).

**Связанные таблицы (активные в _2606):**

| Таблица | Роль |
|---------|------|
| `ipgUtPlP` | Заголовок уточнённого плана (связь с `ipgpPn`); `iuplpLim` — суммарный лимит, fallback при пустых детализациях |
| `ipgUtPlGrP` | Привязка плана к группе планов (`ipgUtPlGr`) |
| `ipgPnLim` | Годовые лимиты по stCost на уровне `ipgPn` (эталон для сверки SUM(месяцев)) |
| `ipgUtPlPnLmMn` | Помесячное плановое освоение по stCost |
| `ipgUtPlPnLmYe` | Годовые лимиты по stCost на уровне уточнённого плана; читается через `fnStCostUtPlP` |

**Не использовать для расчётов _2606:** `ipgUtPlPnCostIts` (устаревшая широкая модель, см. отдельную статью).

**См. также:** группа планов, элемент структуры затрат, накопленный план, помесячный план по stCost.

### Помесячный план по элементу stCost (`ipgUtPlPnLmMn`)

Нормализованное хранилище **планового (планируемого) освоения на месяц** по **пункту уточнённого плана** и **элементу структуры затрат**.

**Модель данных:** одна строка = комбинация `(iuplpmPlPn, iuplpmStCost, iuplpmMn)`; сумма в `iuplpmLim` (в миллионах руб., как в `ipgPnLim`). Хранится **только** при `iuplpmLim > 0` (**sparse**); отсутствие месяца = **нет строки**, не нулевая сумма.

**Ограничение целостности:** `CK_ipgUtPlPnLmMn_gt0` (`iuplpmLim > 0`); включается скриптом `09c` (Решение 15). Аналоги: `ipgUtPlPnLmQu`, `ipgUtPlPnLmYe`. Legacy `CK_*<>0` сняты в `09c`.

| Колонка | Смысл |
|---------|--------|
| `iuplpmPlPn` | FK → `ipgUtPlP.iuplpKey` |
| `iuplpmStCost` | Элемент структуры затрат (`stCost`) |
| `iuplpmMn` | Номер месяца (1…12) |
| `iuplpmLim` | Плановое освоение за месяц (**> 0**) |

**Использование в стеке _2606:** чтение через `ags.fnStCostUtPlPMn` → `fnStCostRsIpgPn_2606` → mastering / `spMstrg_2606`. Приёмка: скрипты `07m`/`07n`/`07o`, fixture `dev-chain5-utpl-stcost/`. Контроль: `SUM(iuplpmLim)` по **существующим** месяцам ≈ лимит `ipgPnLim` (или fallback `ipgpSmTtl` / `ipgpSmWrk` / …) для stCost **212 / 195 / 172 / 187**.

**Не путать с:** `ipgUtPlP.iuplpM01`…`M12` (суммарные месячные колонки **без** разбивки по stCost); `ipgUtPlPnCostIts` (устаревшая широкая таблица); нулевые строки «для заполнения сетки 12×4» в fixture (запрещено с 18.8.2, см. Решение 15).

**Последнее обновление статьи:** 2026-06-26.

### Пункт инвестпрограммы (`ipgPn`, `ipgp`)

Привязка стройки (`cstAgPn`) к конкретной ИП в цепи. Имеет лимиты (`ipgPnLim`, `ipgpSmTtl` / `ipgpSmWrk` / …) и опционально помесячный UtPl. У одной стройки в цепи может быть **несколько** пунктов ИП (по одному на каждый период актуальности разных ИП).

## Р

### Раздел структуры ИПГ (`stIpg`, параметр `@ipgStKey`)

Узел дерева разделов инвестпрограммы (`ags.stIpg`, связь со стройкой через `ipgStPn`). Параметр `@ipgStKey` в `spMstrg_2606` / `fnMasteringStIpgStCost_2606` задаёт **фильтр строек** в цепи: стройка включается, если хотя бы один её `ipgpPn` привязан к узлу `stIpg` или его потомку (`fnStDownAll` + корень).

**Не путать с:** инвестпрограммой в цепи (`ipgChRlV`, смена ИП 6→8→11); с элементом структуры **затрат** `stCost` (`@stCostKey`).

**Внутри стройки** фильтр `@ipgStKey` **не сужает** декомпозицию: после отбора `cstAgPn` стек считает все пункты ИП цепи и помесячный UtPl по stCost. Если стройка прошла фильтр при разных `@ipgStKey`, её строки в отчёте совпадают.

**Последнее обновление статьи:** 2026-06-26.

### Расчёт освоения

Пересчёт планов, фактов и процентов освоения по цепи ИП (форма MS Access `ipgChMin`, зелёная кнопка «%», процедуры и функции стека `_2606`). Включает определение плановых показателей по UtPl, накопленных лимитов и факта по видам документов.

**Не путать с:** ревизией.

### Ревизия

Синхронизация **факта** между Excel и SQL в форме MS Access `ra_a` (загрузка/сверка документов факта). Относится к документам RA, АВ, РАЛП и др., **не** к планам, лимитам UtPl и смене инвестпрограмм в цепи.

## С

### Sparse-хранение UtPl (лимит > 0)

Политика нормализованных таблиц разбивки плана (`ipgUtPlPnLmMn`, `ipgUtPlPnLmQu`, `ipgUtPlPnLmYe`): в БД присутствуют **только** строки с `iuplp*Lim > 0`. Отсутствие планового освоения в периоде = **отсутствие строки**, не нулевое значение.

CHECK-ограничения `CK_ipgUtPlPnLm*_gt0` (`lim > 0`); включение — deploy `09c` (Решение 15, этап 18.8.4). Стек `_2606` при отсутствии строки трактует план как `0`.

**См. также:** помесячный план по stCost, плановые показатели UtPl.

### Смена актуальности инвестпрограммы

Переход от одной ИП к следующей в цепи по дате `ipgcrvStr` следующей записи `ipgChRlV`; дата окончания предыдущей — `ipgcrvEnd` (вычисляемая). В стеке `_2606` учитывается с **дневной** гранулярностью (`fnIpgChDatsV`).

**Не путать с:** ревизией факта.

**Политика плана на переходе (принято):** сохраняется «ужесточение» из `_2605` — на дате смены в накопленный план входит **полный** месячный прирост UtPl за месяц перехода (без пропорции по дням). Корректная интерполяция плана по дням и согласованный cutoff факта по датам документов — **отдельная задача** (см. Решение 14 в `03-design-decisions.md`).

## У

### Устаревшая таблица `ipgUtPlPnCostIts`

**Статус:** устаревшая; **не используется** в стеке `_2606` и кандидат на удаление после согласования с владельцами данных.

Альтернативная **широкая** модель помесячного плана по stCost: одна строка на пару `(план, stCost)` с колонками `iuplpcM01`…`M12` вместо нормализованных строк в `ipgUtPlPnLmMn`.

| | `ipgUtPlPnLmMn` (активная) | `ipgUtPlPnCostIts` (устаревшая) |
|---|---|---|
| Формат | Long: строка на месяц | Wide: M01…M12 в одной строке |
| Ссылка из функций расчёта | ✅ `fnStCostUtPlPMn` | ❌ нет ссылок из функций/процедур |
| Данные на dev (цепь 5) | ~9 708 строк | 0 строк |

При наполнении планов и приёмке `_2606` ориентироваться **только** на `ipgUtPlPnLmMn`. Таблицу `ipgUtPlPnCostIts` не наполнять и не использовать в новых разработках.

**Последнее обновление статьи:** 2026-06-26.

## Ц

### Цепь инвестпрограмм (`ipgCh`)

Упорядоченная последовательность инвестпрограмм для отчётности по освоению (например, цепь **5** — «Газпром, 2022», 680 контрактов, 17 контрольных дат в тестах `_2606`).

## Э

### Элемент структуры затрат (`stCost`)

Узел структуры затрат (`stCost`), по которому разделяются лимиты и план. В приёмке планов UtPl: **212** (итог), **195** (СМР), **172** (оборудование), **187** (прочие).
