## А

### Агентский эталон (agency-golden)

Тестовая стройка (или пара строек) для приёмки **агентской** схемы (`ipgpSh = 2`): динамика `ag_presented` / `ag_accepted`, помесячный и накопленный план (`ag_Pl` / `ag_PlAccum`) при смене ИП **6 / 8 / 11**, декомпозиция UtPl по stCost.

**Принято (Решение 21, цепь 5):** `@ipgStKey = 4`, `cstAgPn` **849** (`051-2000973`) и **1862** (`051-2000860`) — две agency-стройки в одном узле структуры.

**Не использовать для agency-тестов:** invest-golden **2102** (только `ipgpSh = 1`, факт освоения ≈ 0).

**См. также:** инвестиционный эталон, `13-plan-stcost-monthly-acceptance.md` §17–18.

**Последнее обновление статьи:** 2026-06-30.

## В

### Внелимитная стройка (OUT_GROUP)

Стройка (`cstAgPn`) цепи, **не** входящая в IN_GROUP выбранного узла `@ipgStKey` (нет привязки `ipgStPn` к поддереву узла), но попадающая в отчёт, если:

1. её **тип по 5-й литере** кода САК разрешён в `ags.stIpgOutLimPn` для `@ipgStKey`;
2. стройка ∈ **`chainActive`** (7 источников, как `fnIpgChRsltCst` — Решение 16, доп. 2026-06-29).

План и лимит из `ipgPn` могут отсутствовать; **факт** (RA, АВ и др.) отображается в RS по правилам `fn2`.

**Активность:** см. п. 2; отдельный предикат `lim>0` / `accepted>0` в TVF **не** применяется.

**Не путать с:** **неплановыми капзатратами (`np_`)** — OUT_GROUP задаёт **отбор строек** по разделу `stIpg`; `np_` — **раскладка капфакта** по схемам, когда у стройки нет пункта `ipgPn` на данной ИПГ (может быть и у стройки IN_GROUP).

**См. также:** универсум строек отчёта, тип стройки по коду САК, раздел структуры ИПГ, неплановые капзатраты.

**Последнее обновление статьи:** 2026-06-29.

**Dev-fixture (цепь 5):** тестовые группы **18 / 19 / 20** (`TEST_CH5_IPG6/8/11`); prod-подобные группы **3 / 4 / 6** **не изменять** в fixture (Решение 21).

## И

### Инвестпрограмма (ИП)

Версия инвестиционной программы в рамках цепи (`ipgCh`). Имеет срок актуальности в цепи: `ipgcrvStr` … `ipgcrvEnd` в таблице `ipgChRl_2606` (до 21.1: `ipgChRlV`). На конкретную дату расчёта актуальна **не более одной** ИП цепи.

### Инвестиционный эталон (invest-golden)

Тестовая стройка для приёмки **инвестиционной** схемы, календаря RS1 (17 дат), фильтра `stIpg` на листе дерева.

**Принято (Решение 21, цепь 5):** `@ipgStKey = 42` (или **61**), `cstAgPn = 2102`; колонки `iv_*` / `ia_*`; UtPl fixture в группах **18–20**.

**Не путать с:** agency-golden (факт и план по `ag_*`).

**Последнее обновление статьи:** 2026-06-30.

### Связь цепь–инвестпрограмма (`ipgChRl_2606`)

Таблица сроков актуальности ИП в цепи (замена legacy `ipgChRl` по Дефекту Б). Хранит `ipgcrvStr`, `ipgcrvUtPlGr`; `ipgcrvEnd` — вычисляемый столбец (`fnIpgChRlEnd_2606`).

| Объект | Legacy (prod) | Стек `_2606` (целевое имя) |
|--------|---------------|----------------------------|
| Связь цепь↔ИП | `ipgChRl` | **`ipgChRl_2606`** |
| Группа планов на ИП | `ipgcrUtPlGr` | **`ipgcrvUtPlGr`** |

Mastering и приёмка UtPl (`07n`/`07o`) читают **`ipgChRl_2606`**. До этапа **21.2** PercentBrn plan-JOIN ещё может ссылаться на legacy `ipgChRl` — дефект, устраняется патчем `05b`.

**Последнее обновление статьи:** 2026-06-30.

**См. также:** смена актуальности инвестпрограммы, цепь инвестпрограмм, пункт инвестпрограммы.

## К

### Календарь контрольных дат RS1 (`fnIpgChDats_2606`, `dateRslt`)

Упорядоченный набор дат среза для **полного рекордсета RS1** (графики Java/FEMSQ, 01.01–31.12 расчётного года). В стеке `_2606` **канонический** источник — `ags.fnIpgChDats_2606(@ipgCh)` (до 21.1: `fnIpgChDatsV`):

| Компонент | Содержание |
|-----------|------------|
| Начало года | **01.01** (`dAll`) |
| Месяцы | 12 × конец месяца (`EOMONTH`) |
| Смена ИПГ | `ipgcrvStr` / `ipgcrvEnd` из `ipgChRl_2606` |

**Цепь 5, 2022:** **17** дат. PercentBrn / RS1 **обязаны** выдавать те же `dateRslt`, что `fnIpgChDats_2606` (Решение 17).

**Legacy `_2605`:** календарь `@dt` в PercentBrn даёт **16** дат (без 01.01); костыль `mNum=0` в plan-JOIN — **дефект**, не целевая модель.

**Не путать с:** `@MounthEndDate` (фильтр RS4–RS7 для Access); помесячной матрицей fn2 (`mNum` 1…12).

**См. также:** начальная точка графика, накопленный план. `06-sp-recordsets-and-acceptance.md`, `03-design-decisions.md` §17.

**Последнее обновление статьи:** 2026-06-30.

## Н

### Накопленный план

Сумма помесячных плановых показателей UtPl с начала расчётного года до контрольной даты (`dAll` / `@MounthEndDate`). В стеке `_2606` отображается в полях `smmTtl`, `agSmmTtl` и аналогах PercentBrn (`ag_PlAccum`).

На дате окончания периода ИП (`ipgcrvEnd`) накопленный план равен cum UtPl на эту дату и **не обязан** совпадать с годовым лимитом, если ИП снята раньше 31.12.

### Начальная точка графика (01.01)

Контрольная дата **первого января** расчётного года в календаре RS1. Семантика для активной ИПГ на графике освоения:

| Показатель | Ожидание на 01.01 |
|------------|-------------------|
| `ag_Pl`, `ag_PlAccum` | **0** (годовой план ещё не начался) |
| `ag_presented`, `ag_accepted`, … | **0** |
| `ag_lim` | лимит ИПГ на год |
| `ag_restOfLimit` | **≈ лимит** («не освоено и не должно быть освоено») |

Входит в `fnIpgChDats_2606` и mastering; **должна** присутствовать в RS1 после реализации Решения 17. В legacy PercentBrn отсутствовала в `@dt` — добавлялась костылём `mNum=0`.

**Последнее обновление статьи:** 2026-06-30.

### Неплановые капзатраты (`np_`)

**Капитальные** затраты (`typeGr = '1. ОА и Изм.'`) с фактом освоения, но **без пункта `ipgPn`** на данной ИПГ (или без привязки инвестора к цепи). В `fn2` получают `typeGrTtl = '2.2. Агентская, неплан'`, `lim = NULL`, `iShKey = NULL`.

В `PercentBrn` / RS суммы попадают в колонки с префиксом **`np_`** (`np_accepted`, `np_acceptedTtl`, …), **не** в `ag_*`. На **% освоения лимита** (`ag_LimPercent`, `ag_percentDev`) **не влияют**.

Типичные пути в `fn2_2606`: CTE **`extraBase`** (стройка не в `ipgPn` цепи, инвестор в ИПГ) и **`masExtraBase`** (стройка в mastering, но на этой `ipgKey` нет `ipgPn`).

**Не путать с:** внелимитной стройкой (OUT_GROUP) — фильтр по `stIpg`; с **прочими затратами (`oh_`)** — отдельный некапитальный контур.

**См. также:** пункт инвестпрограммы, `09-scheme-cascade-mastering.md` (слой C).

**Последнее обновление статьи:** 2026-06-29.

## П

### Плановые показатели уточнённого плана (UtPl)

Помесячные суммы по пункту инвестпрограммы и элементам структуры затрат.

**Два представления плана (не дублировать без синхронизации):**

| Представление | Где | stCost | Стек `_2606` (расчёт) |
|---------------|-----|--------|------------------------|
| **Широкие колонки** | `ipgUtPlP.iuplpM01`…`M12` | нет колонки; семантика **итога (212)** | **legacy** — только PercentBrn `_2605`/`_2606` (до Решения 22) |
| **Нормализованное** | `ipgUtPlPnLmMn` | да (212/195/172/187…) | **канон** — mastering, `fnStCostRsIpgPn_2606`, К-12/07n |

**Каноническое хранилище помесячной разбивки по stCost для `_2606`:** `ipgUtPlPnLmMn` (см. отдельную статью). Заголовок `ipgUtPlP` остаётся якорем (`iuplpKey`) и связью с группой планов; расчёт `_2606` **не обязан** опираться на `iuplpM*`.

**Связанные таблицы (активные в _2606):**

| Таблица | Роль |
|---------|------|
| `ipgUtPlP` | Заголовок уточнённого плана (связь с `ipgPn` через `iuplpIpgPn`, с группой — через `iuplpPl`); `iuplpLim` — суммарный лимит, fallback при пустых детализациях; см. **широкие месячные колонки** |
| `ipgUtPlGrP` | Привязка плана к группе планов (`ipgUtPlGr`) |
| `ipgPnLim` | Годовые лимиты по stCost на уровне `ipgPn` (эталон для сверки SUM(месяцев)) |
| `ipgUtPlPnLmMn` | Помесячное плановое освоение по stCost |
| `ipgUtPlPnLmYe` | Годовые лимиты по stCost на уровне уточнённого плана; читается через `fnStCostUtPlP` |

**Не использовать для расчётов _2606:** `ipgUtPlPnCostIts` (устаревшая широкая модель, см. отдельную статью).

**См. также:** группа планов, элемент структуры затрат, накопленный план, помесячный план по stCost.

### Помесячный план по элементу stCost (`ipgUtPlPnLmMn`)

Нормализованное хранилище **планового (планируемого) освоения на месяц** по **пункту уточнённого плана** и **элементу структуры затрат**.

**Модель данных:** одна строка = комбинация `(iuplpmPlPn, iuplpmStCost, iuplpmMn)`; сумма в `iuplpmLim` (в миллионах руб., как в `ipgPnLim`). Хранится **только** при `iuplpmLim > 0` (**sparse**); отсутствие месяца = **нет строки**, не нулевая сумма.

**Ограничение целостности:** `CK_ipgUtPlPnLmMn_gt0` (`iuplpmLim > 0`); включается скриптом `09c` (Решение 15). Аналоги: `ipgUtPlPnLmQu`, `ipgUtPlPnLmYe`. Legacy `CK_*<>0` сняты в `09c`.

| Колонка | Смысл |
|---------|--------|
| `iuplpmPlPn` | FK → `ipgUtPlP.iuplpKey` |
| `iuplpmStCost` | Элемент структуры затрат (`stCost`) |
| `iuplpmMn` | Номер месяца (1…12) |
| `iuplpmLim` | Плановое освоение за месяц (**> 0**) |

**Использование в стеке _2606:** чтение через `ags.fnStCostUtPlPMn` → `fnStCostRsIpgPn_2606` → mastering / `spMstrg_2606`. Приёмка: скрипты `07m`/`07n`/`07o`, fixture `dev-chain5-utpl-stcost/`. Контроль: `SUM(iuplpmLim)` по **существующим** месяцам ≈ лимит `ipgPnLim` (или fallback `ipgpSmTtl` / `ipgpSmWrk` / …) для stCost **212 / 195 / 172 / 187**.

**Не путать с:** широкие месячные колонки `ipgUtPlP` (итог без stCost, legacy UI); `ipgUtPlPnCostIts` (устаревшая широкая таблица); нулевые строки «для заполнения сетки 12×4» в fixture (запрещено с 18.8.2, см. Решение 15).

**Последнее обновление статьи:** 2026-07-06.

### Правило внелимитных типов раздела ИПГ (`stIpgOutLimPn`)

Таблица связи **узел `stIpg` → разрешённые типы** внелимитных строек (1→N). Колонки: `siolpStIpg` → `stIpg.stiKey`, `siolpCstType` → `char(1)` (`'1'` ПИР, `'2'` стройка, `'3'` бурение — 5-я литера кода САК).

Примеры: узел **1** — три строки `1`,`2`,`3`; **51** — `2`,`3`; **45** — `1`; листья без строк — только стройки группы (IN_GROUP).

**Не путать с:** `ipgStLm` / `stiLim` (legacy, не в runtime).

**Последнее обновление статьи:** 2026-06-24.

### Прочие затраты в отчёте (`oh_`)

**Некапитальные** затраты по отчётам агентов: `typeGr = '2. ОА, прочие и Изм'`, в отчёте `typeGrTtl = '4. Прочие'`. Всегда `ipgKey = NULL`; контур **вне** mastering (`fnMastering*` не вызывается).

В `PercentBrn` суммы — в колонках **`oh_*`** (оконная агрегация по стройке и месяцу), **не** в `ag_accepted` / `np_accepted`. На лимит и % освоения ИПГ **не влияют**.

Источник строк в `fn2_2606`: CTE **`nullIpgBase`**. При наличии в том же месяце строк с `ipgKey` отдельная строка `4. Прочие` скрывается (`ipgCount > 0`), суммы остаются в `oh_*` на ИПГ-строках.

**Не путать с:** элементом stCost **187** («прочие» в структуре затрат); с **неплановыми капзатратами (`np_`)** — тот же отчёт, но капитальный контур.

**См. также:** неплановые капзатраты, `09-scheme-cascade-mastering.md` (слой D).

**Последнее обновление статьи:** 2026-06-29.

### Пункт инвестпрограммы (`ipgPn`, `ipgp`)

Привязка стройки (`cstAgPn`) к конкретной ИП в цепи. Имеет лимиты (`ipgPnLim`, `ipgpSmTtl` / `ipgpSmWrk` / …) и опционально помесячный UtPl. У одной стройки в цепи может быть **несколько** пунктов ИП (по одному на каждый период актуальности разных ИП).

Отсутствие пункта на **конкретной** ИПГ при наличии факта → **неплановые капзатраты (`np_`)** на этой ИПГ, а не исключение стройки из отчёта.

**Последнее обновление статьи:** 2026-06-29.

## Р

### Раздел структуры ИПГ (`stIpg`, параметр `@ipgStKey`)

Узел дерева разделов инвестпрограммы (`ags.stIpg`, связь со стройкой через `ipgStPn`). Параметр `@ipgStKey` в `spMstrg_2606` задаёт **контекст раздела** для отбора строек.

**IN_GROUP:** стройка включается, если хотя бы один её `ipgpPn` связан с узлом `stIpg` или его потомком (`fnStDownAll` + корень) — как в `fnMasteringStIpgStCost_2606`.

**OUT_GROUP (внелимитные):** стройки вне IN_GROUP, разрешённые таблицей `stIpgOutLimPn` по типу кода и **активности** = членство в `chainActive` (Решение 16, доп. 2026-06-29). У листовых узлов (напр. 42, 61) строк в `stIpgOutLimPn` нет → только IN_GROUP.

**Внутри стройки** фильтр `@ipgStKey` **не сужает** декомпозицию UtPl: после отбора считаются все пункты ИП цепи и помесячный план по stCost.

**Не путать с:** инвестпрограммой в цепи (`ipgChRlV`); с `stCost` (`@stCostKey`); с legacy `ipgStLm`/`stiLim` (не используются в runtime).

**Последнее обновление статьи:** 2026-06-24.

### Расчёт освоения

Пересчёт планов, фактов и процентов освоения по цепи ИП (форма MS Access `ipgChMin`, зелёная кнопка «%», процедуры и функции стека `_2606`). Включает определение плановых показателей по UtPl, накопленных лимитов и факта по видам документов.

**Не путать с:** ревизией.

### Ревизия

Синхронизация **факта** между Excel и SQL в форме MS Access `ra_a` (загрузка/сверка документов факта). Относится к документам RA, АВ, РАЛП и др., **не** к планам, лимитам UtPl и смене инвестпрограмм в цепи.

## С

### Sparse-хранение UtPl (лимит > 0)

Политика нормализованных таблиц разбивки плана (`ipgUtPlPnLmMn`, `ipgUtPlPnLmQu`, `ipgUtPlPnLmYe`): в БД присутствуют **только** строки с `iuplp*Lim > 0`. Отсутствие планового освоения в периоде = **отсутствие строки**, не нулевое значение.

CHECK-ограничения `CK_ipgUtPlPnLm*_gt0` (`lim > 0`); включение — deploy `09c` (Решение 15, этап 18.8.4). Стек `_2606` при отсутствии строки трактует план как `0`.

**См. также:** помесячный план по stCost, плановые показатели UtPl.

### Смена актуальности инвестпрограммы

Переход от одной ИП к следующей в цепи по дате `ipgcrvStr` следующей записи `ipgChRl_2606`; дата окончания предыдущей — `ipgcrvEnd` (вычисляемая). В стеке `_2606` учитывается с **дневной** гранулярностью (`fnIpgChDats_2606`).

**Не путать с:** ревизией факта.

**Политика плана на переходе (принято):** сохраняется «ужесточение» из `_2605` — на дате смены в накопленный план входит **полный** месячный прирост UtPl за месяц перехода (без пропорции по дням). Корректная интерполяция плана по дням и согласованный cutoff факта по датам документов — **отдельная задача** (см. Решение 14 в `03-design-decisions.md`).

## Ш

### Широкие месячные колонки `ipgUtPlP` (`iuplpM01`…`M12`)

Помесячное плановое освоение в **денормализованном** виде: по одной паре колонок на месяц (`iuplpMnn` — прирост за месяц, `iuplpMnnAccum` — накопленный итог с начала года). Хранятся на **заголовке** уточнённого плана (`ipgUtPlP`), без отдельной колонки `stCost`.

**Семантика уровня stCost:** колонки **не разбивают** план по 195/172/187; по смыслу соответствуют **суммарному плану уровня итога структуры затрат — stCost 212** («всего по объекту»), как `ipgpSmTtl` / `ipgPnLim@212`. Декомпозиция на подвиды — только в `ipgUtPlPnLmMn`. Сверка: К-12 сравнивает `SUM(iuplpmLim)` при `iuplpmStCost=212` с лимитом 212; К-13 — равенство 212 = 195+172+187 **в LmMn**, не в `iuplpM*`.

**Единицы:** миллионы рублей (как в `ipgUtPlPnLmMn`); в PercentBrn умножаются на `1 000 000`.

| Аспект | `_2605` / legacy PercentBrn | `_2606` (mastering) |
|--------|----------------------------|---------------------|
| Чтение `iuplpM*` | **да** — подзапросы gap/gip/gup | **нет** |
| Чтение `ipgUtPlPnLmMn` | косвенно / не для RS-колонок плана | **да** — `fnStCostUtPlPMn` → `fnStCostRsIpgPn_2606` |
| Sparse UtPl (Решение 15) | prod: часть строк с заполненными `M*`; dev fixture gr 18–20: часто **только LmMn** | отсутствие строки LmMn = план `0` |

**Связь с группой планов:** `ipgUtPlP.iuplpPl` → `ipgUtPlGrP` → `ipgUtPlGr`; активная группа на ИП в цепи — `ipgChRl_2606.ipgcrvUtPlGr`. У одного `ipgPn` может быть **несколько** строк `ipgUtPlP` (разные `iuplpPl` / группы); LmMn привязан к конкретному `iuplpKey`.

**Связь `iuplpSubAg` / og на заголовке:** историческая привязка к филиалу агента; после смены филиала в рамках одной ИП практическая ценность для отчёта снижена (вне scope расчёта плана в `_2606`).

**Диагностика 21.4.2:** при заполненном LmMn и пустых `iuplpM*` mastering и К-12 проходят, а `ag_Pl`/`iv_Pl` в PercentBrn — нет (до реализации Решения 22).

**См. также:** плановые показатели UtPl, помесячный план по stCost, `03-design-decisions.md` §22, `diag-21_4_2-plan-align-findings.md`.

**Последнее обновление статьи:** 2026-07-06.

## Ф

### Фантомный уровень филиала (GROUPING SETS)

Артефакт legacy `GROUPING SETS` в PercentBrn: при отсутствии филиала у стройки создаётся уровень агрегации `(ogNm, branch, dateRslt)` с **`branch = 0`**, дублирующий уровень агента в **RS1**. В производных RS4–RS7 такие строки отфильтрованы (`cstAgPnCode='всего' AND branch=0`), в RS1 остаются.

**Не путать с:** реальным филиалом контрагента; с календарём 17 дат (Решение 17).

**Статус _2606:** известный дефект; устранение — отдельное решение (этап 20.6 чат-плана), не блокер gate **parity** с `_2605`.

## У

### Универсум строек отчёта (IN_GROUP ∪ OUT_GROUP)

Множество строек (`cstAgPn`), попадающих в `fn2_2606` / `spMstrg_2606` при заданном `@ipgStKey`:

| Компонент | Определение |
|-----------|-------------|
| **IN_GROUP** | стройки с привязкой `ipgStPn` к `@ipgStKey` или потомку (`fnStDownAll` + корень) |
| **OUT_GROUP** | стройки цепи **вне** IN_GROUP, разрешённые `stIpgOutLimPn` по типу (`fnCstAgPnTypeChar`) и **`chainActive`** (7 источников, как `fnIpgChRsltCst`) |

При `@ipgStKey = NULL` — вся цепь (без сужения). При листовых узлах без строк в `stIpgOutLimPn` (напр. 42, 61) — только IN_GROUP.

Фильтр универсума **не меняет** раскладку факта по схемам внутри стройки: капфакт без `ipgPn` → `np_*`, некапитальные → `oh_*` (см. отдельные статьи).

**См. также:** внелимитная стройка, раздел структуры ИПГ, неплановые капзатраты, прочие затраты в отчёте. Решение 16 в `03-design-decisions.md`.

**Последнее обновление статьи:** 2026-06-29.

### Устаревшая таблица `ipgUtPlPnCostIts`

**Статус:** устаревшая; **не используется** в стеке `_2606` и кандидат на удаление после согласования с владельцами данных.

Альтернативная **широкая** модель помесячного плана по stCost: одна строка на пару `(план, stCost)` с колонками `iuplpcM01`…`M12` вместо нормализованных строк в `ipgUtPlPnLmMn`.

| | `ipgUtPlPnLmMn` (активная) | `ipgUtPlPnCostIts` (устаревшая) |
|---|---|---|
| Формат | Long: строка на месяц | Wide: M01…M12 в одной строке |
| Ссылка из функций расчёта | ✅ `fnStCostUtPlPMn` | ❌ нет ссылок из функций/процедур |
| Данные на dev (цепь 5) | ~9 708 строк | 0 строк |

При наполнении планов и приёмке `_2606` ориентироваться **только** на `ipgUtPlPnLmMn`. Таблицу `ipgUtPlPnCostIts` не наполнять и не использовать в новых разработках.
