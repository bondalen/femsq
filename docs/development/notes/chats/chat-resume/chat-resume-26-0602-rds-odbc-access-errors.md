# Chat Resume: 26-0602 — ошибки ODBC/SQL Server в MS Access через RDS (Basis Workplace)

## Метаданные

- **Дата:** 2026-06-01 – 2026-06-02
- **Тема:** периодические ошибки подключения MS Access к SQL Server (`FishEye` / `SPB-05-NV-SQL1`) при работе через удалённый рабочий стол (Basis Workplace, RDS-хост `NV-SK-TSW112`)
- **Ограничение чата:** анализ и предложения; **изменения в коде, DSN и конфигурации сервера не вносились**
- **Базовая линия сеанса (02.06.2026):** пользователь `bondale1`, Session ID **153**, `rdp-tcp#8`, вход 08:40, `Test-NetConnection` к порту 1433 — успешно (ошибка на момент снимка не воспроизводилась)

## Контекст

- Клиент: **MS Access** (связанные таблицы и pass-through через User DSN **`FishEye`**).
- Сервер БД: **`SPB-05-NV-SQL1`**, база **`FishEye`**, драйвер **SQL Server Native Client 11.0**.
- Доступ к Windows: **Basis Workplace** → RDS на **`NV-SK-TSW112.adm.gazprom.ru`** (много пользовательских сеансов).
- Симптомы: ошибки не постоянные; после «Отключить» в Basis ошибка часто **сохраняется**; иногда исчезает сама (таймаут сеанса / другой хост пула).

## Итог (диагноз)

**Корневая причина** — исчерпание эфемерных TCP-портов на RDS-хосте (**Event ID 4227**, источник `Tcpip`), а не отказ SQL Server или «сломанный» DSN как таковой.

Цепочка:

1. VBA и ODBC часто открывают/закрывают TCP-соединения (особенно `btnAuditRun_Click` → `ra_aAgFee23_06.Audit` при `af_type = 6`).
2. Порты остаются в **TIME_WAIT** (~240 с по умолчанию).
3. На хосте с десятками сеансов порты заканчиваются → **`WSAEADDRNOTAVAIL`** («Требуемый адрес для своего контекста неверен»).
4. Падают и ODBC-тест DSN, и `telnet`/`Test-NetConnection` **из той же сессии** — это признак деградации сетевого стека сессии, а не SQL Server.
5. Basis при «Отключить» возвращает в **тот же Session ID** → состояние сохраняется; радикальное снятие — **log off** (завершение сеанса).

**Дополнительно:** попытка DSN с `tcp:SPB-05-NV-SQL1,1433` при уже деградированном стеке даёт ту же TCP-ошибку; `netsh winsock show catalog` — каталог чистый (LSP не виноваты).

## Предложения по устранению (кратко)

### Пользователь / без прав администратора

- Перед длинной ревизией: зафиксировать Session ID (`query user`); при ошибке сравнить с **153**.
- Снятие ошибки: **log off** сеанса (не «Отключить» в Basis); при отсутствии пункта — `logoff` в cmd или сброс сеанса администратором.
- Профилактика: не гонять ревизию подряд; проверять `Test-NetConnection SPB-05-NV-SQL1 -Port 1433`.
- VBA: см. `vba-fix-proposal.md` (кэш ADODB, retry, закрытие QueryDef; отдельно — оптимизация `ra_aAgFee23_06`).

### Администратор RDS / Basis

- `TcpTimedWaitDelay = 30`, расширение dynamic port range, ODBC connection pooling для SNAC 11 (32-bit).
- Таймаут завершения disconnected-сессий в Basis/GPO.
- Расследование ночной нагрузки (~22:00) по Event 4227.
- Закрепление пользователя за постоянным хостом пула (сохранность профиля и DSN).
