## Единый стек `_2605` (продуктив)

Все семь рекордсетов происходят из **одной** временной таблицы `@TableFnIpgChRsltCstUtlPercentBrn_*`, заполняемой функцией `fnIpgChRsltCstUtlPercentBrn_2605` → `fnIpgChRsltCstUtl2_2605` → `fnIpgChRsltCstUtl_2408`.

```
fnIpgChRsltCstUtl_2408          — факт по (год, месяц, стройка); лимит per ipgKey
        ↓
fnIpgChRsltCstUtl2_2605         — месячная матрица + MONTH-фильтр строк ИПГ
        ↓
fnIpgChRsltCstUtlPercentBrn_2605 — @dt / dateRslt, планы (iuplpM*), проценты
        ↓
@TableFnIpgChRsltCstUtlPercentBrn_2605
        ↓
spMstrg_2605 → ResultSet1..7
```

**Не два параллельных стека** — одна цепочка с разными уровнями агрегации и последующей фильтрацией.

## Семантика данных (установлено 2026-06-09)

### Факт (`presented`, ОПИ, RA…)

- На уровне **fn2 / _2408**: месячный итог по `(год, месяц, cstAgPnKey)` **без привязки к ipgKey**.
- На каждую «активную в месяце» ИПГ (MONTH-фильтр) реплицируется **один и тот же** месячный факт.
- На уровне **PercentBrn / dateRslt**: для переходных дат (21.09, 22.09) **тот же** `ag_presented`, но **разные** `ag_lim` / `ag_Pl` → разный `ag_percentDev`.

**Смысл смены ИПГ в цепи:** корректировка лимитов и планов; при том же факте агентов процент представления меняется (план снижен → % выше).

### Планы и лимиты

- **Per `ipgKey`** из `ipgPn` / `ipgUtPlP` (iuplpM01…M12, Accum).
- Скачок на графике между 21.09 и 22.09 — смена плана/лимита, не неоднозначность факта по дням.

### `dateRslt` в PercentBrn

Таблица `@dt` назначает дату среза на каждую пару `(месяц, ipgKey)`:

| Ситуация в месяце | `dateRslt` |
|-------------------|------------|
| ИПГ **начинается** в месяце | `ipg.ipgStr` |
| ИПГ **заканчивается** в месяце | `ipg.ipgEnd` |
| Иначе | `EOMONTH` месяца |

Дополнительно union всех концов месяцев с `ipgKey` последнего перехода в месяце.

**Цепь 5, 2022** (FishEye): 16 точек `dateRslt`, 14 447 строк PercentBrn; переходы 26.04 / 27.04, 21.09 / 22.09 видны в данных.

## Семь рекордсетов: иерархия

| RS | Назначение | Объём (цепь 5) | Клиент | Фильтр `dateRslt` |
|----|------------|----------------|--------|-------------------|
| **1** | **Полный набор** PercentBrn (~179 полей) | **~14 447** | **Java / FEMSQ** (графики 01.01–31.12) | **Нет** — все точки года |
| 2 | Те же данные, порядок столбцов (ag_, iv_, ia_ вперёд) | = RS1 | Вспомогательный / Access* | Нет |
| 3 | Те же данные, столбцы без ag_/iv_/ia_ | = RS1 | Вспомогательный / Access* | Нет |
| **4** | Сводка **3 месяцев**: текущий + 2 предыдущих | ~сотни | Access (отчёт за месяц) | **`= @MounthEndDate`** (+ PrM, PrM2) |
| 5 | RS4, только `cstAgPnCode='всего'` + UNION | ~десятки | Access | `@MounthEndDate` |
| 6 | Источники освоения (RA, АВ, РАЛП…) | `@MounthEndDate` | Access / отчёты | **`= @MounthEndDate`** |
| 7 | RS6, `cstAgPnCode='всего'` + UNION | ~десятки | Access | `@MounthEndDate` |

\* MS Access **не выводит** RS1 целиком из‑за лимита числа полей (~255). Access использует **производные** RS4–RS7. Java/FEMSQ потребляет **полный RS1** (таблица `spMstrg_*_ResultSet1` или аналог).

**`@MounthEndDate`** — параметр **производных** рекордсетов (текущий месяц и два предыдущих). **Не** ограничивает RS1.

## Контракт приёмки `_2606` ↔ `_2605`

### Приоритет 1 — полный рекордсет (Java)

**Главный критерий:** `spMstrg_2606_ResultSet1` = `spMstrg_2605_ResultSet1` (или эквивалент: полный `fnIpgChRsltCstUtlPercentBrn_2606` = `_2605`).

Проверки:

- `COUNT(*)` и диапазон `dateRslt` (для цепи 5: 2022-01-31 … 2022-12-31, 16 дат).
- Поключевые поля на **всех** `dateRslt` перехода: `ag_presented`, `ag_lim`, `ag_Pl`, `ag_percentDev`, `ipgKey`.
- На 21.09 vs 22.09: равный `ag_presented`, разный `ag_lim` (модель подтверждена: 291/291 пар).

### Приоритет 2 — промежуточные уровни стека

| Уровень | Объект | Назначение в приёмке |
|---------|--------|----------------------|
| A | `fn2_2606` vs `fn2_2605` | Месячный факт + lim per ipgKey (вход PercentBrn) |
| B | `PercentBrn_2606` vs `_2605` | Полный годовой набор → будущий RS1 |

### Приоритет 3 — производные RS (Access)

RS2–RS7 — **производные** от RS1; при совпадении RS1 и идентичной логике sp достаточно spot-check RS4–RS7 при `@MounthEndDate`.

Тестовые параметры цепи 5:

```sql
@ipgCh = 5
@MounthEndDate = '2022-12-31'   -- для RS4–RS7 (окт–ноя–дек 2022)
-- RS1: без фильтра по дате, весь 2022 (~14 447 строк)
```

## Следствия для разработки `_2606`

1. **Не останавливаться на fn2** — приёмка по **полному PercentBrn / RS1**.
2. **PercentBrn_2606** должен воспроизвести `@dt` / все `dateRslt` года, не только month-end.
3. **`@MounthEndDate`** вносить только на этапе sp (RS4–RS7), как в `_2605`.
4. Расхождение fn2 в m9 (305 строк) — **исправлено** CTE `raFact2408` в fn2_2606 (2026-06-09); полный `07e` — перепрогон; при отладке — `07e2` по одной стройке.
