## Единый стек `_2605` (продуктив)

Все семь рекордсетов происходят из **одной** временной таблицы `@TableFnIpgChRsltCstUtlPercentBrn_*`, заполняемой функцией `fnIpgChRsltCstUtlPercentBrn_2605` → `fnIpgChRsltCstUtl2_2605` → `fnIpgChRsltCstUtl_2408`.

```
fnIpgChRsltCstUtl_2408          — факт по (год, месяц, стройка); лимит per ipgKey
        ↓
fnIpgChRsltCstUtl2_2605         — месячная матрица + MONTH-фильтр строк ИПГ
        ↓
fnIpgChRsltCstUtlPercentBrn_2605 — @dt / dateRslt, планы (iuplpM*), проценты
        ↓
@TableFnIpgChRsltCstUtlPercentBrn_2605
        ↓
spMstrg_2605 → ResultSet1..7
```

**Не два параллельных стека** — одна цепочка с разными уровнями агрегации и последующей фильтрацией.

## Семантика данных (установлено 2026-06-09)

### Факт (`presented`, ОПИ, RA…)

- На уровне **fn2 / _2408**: месячный итог по `(год, месяц, cstAgPnKey)` **без привязки к ipgKey**.
- На каждую «активную в месяце» ИПГ (MONTH-фильтр) реплицируется **один и тот же** месячный факт.
- На уровне **PercentBrn / dateRslt**: для переходных дат (21.09, 22.09) **тот же** `ag_presented`, но **разные** `ag_lim` / `ag_Pl` → разный `ag_percentDev`.

**Смысл смены ИПГ в цепи:** корректировка лимитов и планов; при том же факте агентов процент представления меняется (план снижен → % выше).

### Планы и лимиты

- **Per `ipgKey`** из `ipgPn` / `ipgUtPlP` (iuplpM01…M12, Accum).
- Скачок на графике между 21.09 и 22.09 — смена плана/лимита, не неоднозначность факта по дням.

### `dateRslt` в PercentBrn

#### Целевой контракт `_2606` (Решение 17)

Календарь RS1 **должен совпадать** с `fnIpgChDats_2606(@ipgCh)` (до этапа 21.1 в БД: `fnIpgChDats_2606`):

| Компонент | Содержание |
|-----------|------------|
| Начальная точка | **01.01** расчётного года (план=0, факт=0, остаток лимита ≈ лимит) |
| Месяцы | 12 × `EOMONTH` |
| Переходы ИПГ | `ipgcrvStr` / `ipgcrvEnd` из `ipgChRl_2606` (дневная гранулярность) |

**Цепь 5, 2022:** **17** distinct `dateRslt`; полный PercentBrn / RS1 — **больше** legacy ~14 447 строк (добавляются строки на 01.01 × стройки × 4 уровня GROUPING SETS).

#### Текущее поведение legacy `_2605` / PercentBrn_2606 (известный дефект)

Таблица `@dt` в унаследованном коде назначает дату среза на пару `(месяц, ipgKey)`:

| Ситуация в месяце | `dateRslt` |
|-------------------|------------|
| ИПГ **начинается** в месяце | `ipg.ipgStr` |
| ИПГ **заканчивается** в месяце | `ipg.ipgEnd` |
| Иначе | `EOMONTH` месяца |

Дополнительно union концов месяцев с `ipgKey` последнего перехода в месяце. **01.01 в `@dt` нет**; поздний костыль добавляет `DATEFROMPARTS(year,1,1)` в plan-JOIN с **`mNum = 0`** — не единый календарь.

**Наблюдение dev (до fix Решения 17):** **16** точек `dateRslt`, ~14 447 строк; переходы 26.04 / 27.04, 21.09 / 22.09 видны; **2022-01-01 отсутствует** в RS1.

#### GROUPING SETS в RS1

На одну стройку и одну `dateRslt` — **4** уровня агрегации: всего / агент / филиал / деталь. Spot-check `cstAgPn=2102`, `@ipgStKey=42`: **64** = 16×4 (legacy) → целевое **68** = 17×4 после fix.

## Семь рекордсетов: иерархия

| RS | Назначение | Объём (цепь 5) | Клиент | Фильтр `dateRslt` |
|----|------------|----------------|--------|-------------------|
| **1** | **Полный набор** PercentBrn (~179 полей) | **~14 447** (legacy, 16 дат); **целевое:** 17 дат | **Java / FEMSQ** (графики 01.01–31.12) | **Нет** — все точки года |
| 2 | Те же данные, порядок столбцов (ag_, iv_, ia_ вперёд) | = RS1 | Вспомогательный / Access* | Нет |
| 3 | Те же данные, столбцы без ag_/iv_/ia_ | = RS1 | Вспомогательный / Access* | Нет |
| **4** | Сводка **3 месяцев**: текущий + 2 предыдущих | ~сотни | Access (отчёт за месяц) | **`= @MounthEndDate`** (+ PrM, PrM2) |
| 5 | RS4, только `cstAgPnCode='всего'` + UNION | ~десятки | Access | `@MounthEndDate` |
| 6 | Источники освоения (RA, АВ, РАЛП…) | `@MounthEndDate` | Access / отчёты | **`= @MounthEndDate`** |
| 7 | RS6, `cstAgPnCode='всего'` + UNION | ~десятки | Access | `@MounthEndDate` |

\* MS Access **не выводит** RS1 целиком из‑за лимита числа полей (~255). Access использует **производные** RS4–RS7. Java/FEMSQ потребляет **полный RS1** (таблица `spMstrg_*_ResultSet1` или аналог).

**`@MounthEndDate`** — параметр **производных** рекордсетов (текущий месяц и два предыдущих). **Не** ограничивает RS1.

## Контракт приёмки `_2606` ↔ `_2605`

### Три gate (Решение 17 + Решение 21)

| Gate | Назначение | Критерий PASS | Скрипт |
|------|------------|---------------|--------|
| **parity** | Регрессия vs prod `_2605` до релиза fix календаря | Полное совпадение RS1 `_2606` ↔ `_2605` (**все** поля, вкл. `rowNum`, `limSort`); **16** `dateRslt` на цепи 5 | `07s_rs1_parity_chain5.sql` |
| **calendar** | Контракт графика Java (17 дат) | `fnIpgChDats_2606 EXCEPT dateRslt(RS1)` = ∅; на **01.01**: plan=0, rest≈lim; COUNT spot 2102: **68** (=17×4) | `07s_calendar_chain5.sql` |
| **plan-align** | План PercentBrn = mastering / fixture | spot **2102**: `iv_Pl` на ИП **6, 8, 11**; источник — `ipgChRl_2606`, группы **18–20** | этап **21.2** / `07o` |
| **agency-spot** | Agency-golden | `@ipgStKey=4`, cst **849**/**1862**: `ag_lim` + `ag_accepted` на ИП 6/8/11 | этап **21.3** ✅ `07t` |

Gate **parity** обязателен перед флешкой этапа 19. Gate **calendar** — после рефакторе `05` PercentBrn (этап 20). Состояние «parity PASS + calendar FAIL» — **ожидаемо** до fix.

### Приоритет 1 — полный рекордсет (Java)

**До fix календаря (gate parity):** `spMstrg_2606_ResultSet1` = `spMstrg_2605_ResultSet1` (полный `fnIpgChRsltCstUtlPercentBrn_2606` = `_2605`).

**После fix календаря (gate calendar):** `_2606` **намеренно расходится** с `_2605` по COUNT и наличию 01.01; эталон — `fnIpgChDats_2606` + семантика начальной точки, не паритет с prod.

**После plan-align (Решение 21):** расхождение `_2606` ↔ `_2605` по полям плана (`ag_Pl`, `iv_Pl`) на общих датах **допустимо и ожидаемо** — `_2605` читает legacy `ipgChRl`; `_2606` — `ipgChRl_2606` и тестовые группы UtPl **18–20**. Gate **parity** фиксирует состояние до plan-align; после 21.2 — отдельная фиксация в `07s`.

Проверки (parity, текущий prod-like стек):

- `COUNT(*)` и диапазон `dateRslt` (цепь 5: **16** дат, 2022-01-31 … 2022-12-31, без 01.01).
- Все поля построчно по ключу `(dateRslt, ogNm, branch, cstAgPnCode, cstapKey, ipgKey, rowNum, limSort)` для spot `cstAgPn=2102`.
- Поключевые поля на **всех** `dateRslt` перехода: `ag_presented`, `ag_lim`, `ag_Pl`, `ag_percentDev`, `ipgKey`.
- На 21.09 vs 22.09: равный `ag_presented`, разный `ag_lim` (модель подтверждена: 291/291 пар).

Проверки (calendar, после fix):

- **17** distinct `dateRslt`; наличие **2022-01-01**.
- На 01.01: `ag_Pl`/`ag_PlAccum` = 0; `ag_restOfLimit` ≈ `ag_lim`.

### Приоритет 2 — промежуточные уровни стека

| Уровень | Объект | Назначение в приёмке |
|---------|--------|----------------------|
| A | `fn2_2606` vs `fn2_2605` | Месячный факт + lim per ipgKey (вход PercentBrn) |
| B | `PercentBrn_2606` vs `_2605` | Полный годовой набор → будущий RS1 |

### Приоритет 3 — производные RS (Access)

RS2–RS7 — **производные** от RS1; при совпадении RS1 и идентичной логике sp достаточно spot-check RS4–RS7 при `@MounthEndDate`.

Тестовые параметры цепи 5:

```sql
@ipgCh = 5
@MounthEndDate = '2022-12-31'   -- для RS4–RS7 (окт–ноя–дек 2022)
-- RS1: без фильтра по дате, весь 2022 (~14 447 строк)
```

## Следствия для разработки `_2606`

1. **Не останавливаться на fn2** — приёмка по **полному PercentBrn / RS1**.
2. **PercentBrn_2606** должен использовать **`fnIpgChDats_2606`** для `@dt` / `dateRslt` (17 дат, вкл. 01.01), не legacy `@dt` + `mNum=0` (Решение 17).
3. **Plan-JOIN** PercentBrn — **`ipgChRl_2606`**, не legacy `ipgChRl` (группы 3/4/6); fixture только в **18–20** (Решение 21).
4. **`@MounthEndDate`** вносить только на этапе sp (RS4–RS7), как в `_2605`.
5. Расхождение fn2 в m9 (305 строк) — **исправлено** CTE `raFact2408` в fn2_2606 (2026-06-09); полный `07e` — перепрогон; при отладке — `07e2` по одной стройке.
6. **Фантомный филиал** (`branch=0` дублирует агента в RS1) — отдельное решение; не смешивать с календарём (этап 20.6).
7. **Две golden-стройки:** invest **2102** @ `stIpg=42`; agency **849**/**1862** @ `stIpg=4` (Решение 21).
