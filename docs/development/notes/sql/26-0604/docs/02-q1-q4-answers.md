## Q1: `ipg.ipgStRlSh` — что это?

**Вопрос:** `ipg.ipgStRlSh` — ключ схемы `stNet`, ключ ребра `stRel`, или «схема реализации»?

**Ответ: ключ схемы `ags.stNet` (схема связей структуры ИПГ).**

```sql
-- FK: ipg.ipgStRlSh → stNet.stnKey (подтверждено через sys.foreign_keys)
-- Также зеркалится в ags.ipgStRlSh.ipgstrshKey (устаревшая таблица-синоним)

SELECT i.ipgKey, i.ipgNm, i.ipgStRlSh, sn.stnName
FROM ags.ipg i JOIN ags.stNet sn ON sn.stnKey = i.ipgStRlSh
WHERE i.ipgKey IN (25, 26)
-- ipg=25 «Одобренная»:   ipgStRlSh=2, «2021-12, декабрь. (Утв. 2022 года)»
-- ipg=26 «Письмо Д644»:  ipgStRlSh=2, та же схема
```

**Важно:** В контексте _2606 каноничный источник `@stNet` — **`ipgCh.ipgcStNetIpg`** (для цепи 15 = 2), не `ipg.ipgStRlSh`. Оба равны 2 для цепи 15, но `ipgCh.ipgcStNetIpg` — это атрибут цепи и используется во всех существующих функциях (`fnMasteringStIpgStCost`, `fnStCostRsStIpg`).

**Паттерн DAG-фильтра в `_2606`:**
```sql
AND (
    @ipgStKey IS NULL
    OR sp.ipgspSt IN (
        SELECT @ipgStKey                       -- корень (сам узел)
        UNION
        SELECT f.strChild                      -- потомки
        FROM ags.fnStDownAll(ch.ipgcStNetIpg, @ipgStKey) f
        -- ch = JOIN ags.ipgCh ch ON ch.ipgcKey = @ipgChKey
    )
)
```

**Важно (семантика фильтра):** паттерн выше отбирает **стройки** (`cstAgPn`). После отбора `fnMasteringStIpgStCost_2606` считает освоение **по всей стройке** — все пункты ИП цепи и sparse UtPl по месяцам/stCost. Разные `@ipgStKey`, через которые одна и та же стройка попала в выборку, **не меняют** её строки в `fn2` / `spMstrg_2606` (dev: `cstAgPn = 2102`, `stIpg` 42 vs 61 — `EXCEPT` = 0). Узлы `stIpg` ≠ инвестпрограммы цепи (`ipgChRl_2606`). См. `03-design-decisions.md`, [`glossary.md`](../../../../../project/glossary.md).

**Что НЕ использовать:**
- `ags.fnIpgStNum` — работает с устаревшим семейством `ipgSt` (изолировано от ipgPn с 2021-05-31)
- `ags.fnSt` напрямую — в кодовой базе не используется в таких контекстах; `fnStDownAll` + UNION — устоявший паттерн

## Q2: Интерфейс клиента — `@ipgStKey int` или `@ipgSt nvarchar`?

**Вопрос:** передавать `@ipgStKey int` (числовой ключ узла `stIpg`) или `@ipgSt nvarchar` с lookup по `stIpgNm`?

**Ответ: `@ipgStKey int`.**

Обоснование:
- `@ipgSt nvarchar` в `_2605` — это код `importIpgSt_26-0320.cst_type` (значения: `11ПРЗ`, `12ОПР`, `20ПКП`, `23КПР`, `30ППР`, `99ОСТ`). Это **коды типов затрат**, не структура ИПГ. Два параметра несопоставимы по смыслу.
- `@ipgStKey int` соответствует `stIpg.stiKey` / `st.stKey`. Пример: 21 = «Объекты добычи газа».
- Access/VBA легко передаёт целое число.
- `_2606` — независимая версия, обратная совместимость с `_2605` не требуется.

```sql
-- Пример lookup для Access: выбрать ключ по имени
SELECT i.stiKey, n.stinNm
FROM ags.stIpg i
JOIN ags.stIpgNm n ON n.stinIpg = i.stiKey
ORDER BY n.stinNm
```

## Q3: Нужен ли `@stCostKey int` в `spMstrg_2606`?

**Вопрос:** Нужен ли параллельный фильтр по структуре затрат (`@stCostKey int`) в `spMstrg_2606`, или это отдельный этап?

**Первоначальный ответ** (в ходе Q3): отдельный этап.

**Ответ пересмотрен 2026-06-04 — `@stCostKey` включён в `_2606`.**

**Причины пересмотра:**

1. В кодовой базе уже существует стек `fnMasteringStIpgStCost` (с `fnStCostRsIpgPn`, `fnStCostRsCstAgPn`, `fnMasteringCstAgPnSh`), реализующий оба параметра. `spMstrg_2605` — частный случай при `@ipgStKey=NULL, @stCostKey=NULL`.

2. Создание `_2606` в два прохода (сначала без `@stCostKey`, потом с ним) — значительно трудоёмче, чем сразу на стеке A.

3. `fnMasteringStIpgStCost` содержит встроенный Дефект А (всегда берёт `max(c.ipgcStNetIpg)`) — его исправление требует версионированной копии в любом случае.

**Принятое решение:** строить `_2606` на стеке `fnMasteringStIpgStCost`; оба параметра `@ipgStKey int` и `@stCostKey int` — nullable; для цепи 15 `@stCostKey` работает немедленно через fallback к `ipgUtPlP.iuplpLim`.

**Детали:** `docs/development/notes/sql/26-0604/docs/03-design-decisions.md §7`.

## Q4: `ipgcrvUtPlGr` для ipg=26 в таблице `ipgChRl_2606`

**Вопрос:** Для цепи 15, значение у `ipg=25` равно 17 (из `ipgChRl`). Для `ipg=26` — NULL. Уточнить корректное значение.

**Ответ: NULL — это корректное значение.**

```sql
-- ipgChRl для цепи 15:
-- ipgcrKey=33: ipgcrChain=15, ipgcrIpg=25, ipgcrUtPlGr=17
-- ipgcrKey=34: ipgcrChain=15, ipgcrIpg=26, ipgcrUtPlGr=NULL

-- В ags.ipgUtPlGr для ipg=26 нет записи (последняя — key=17 для ipg=25)
SELECT * FROM ags.ipgUtPlGr ORDER BY iuplgKey
-- Всего 17 записей, max iuplgIpg=25
```

«Письмо Д644» (ipg=26) не имеет выделенной группы планов. `ipgcrvUtPlGr = NULL` в новой таблице `ipgChRl_2606` для этой строки — семантически корректно. Функции, работающие с лимитами через `ipgUtPlGr`, должны обрабатывать NULL (лимиты по ipg=26 недоступны).

## Этап 0.2 — Тест DAG-обхода

```sql
-- Узел 21 «Объекты добычи газа» — листовой в stNet=2
SELECT * FROM ags.fnSt(2, 21)
-- Возвращает только сам узел 21 (parent=16, level=0)

SELECT * FROM ags.fnStDownAll(2, 21)
-- Возвращает 0 строк (нет потомков, корень исключён)

-- Покрытие ipgStPn для узла 21:
SELECT p.ipgpIpg, COUNT(*) AS cnt_ipgStPn
FROM ags.ipgStPn sp JOIN ags.ipgPn p ON p.ipgpKey = sp.ipgspPn
WHERE sp.ipgspSt IN (SELECT ipgsrChaild FROM ags.fnIpgStNum(2, 21))
  AND p.ipgpIpg IN (25, 26)
GROUP BY p.ipgpIpg
-- ipg=25: 25 пунктов; ipg=26: 32 пункта

-- Уникальных cstAgPnCode (union ipg=25+26): 34 стройки
```

## Этап 0.5 — Выбор тестовой цепи

**Контекст:** цепь 15 не подходит для разработки основного стека — `ipgPnLim` и `ipgUtPlPnLmMn` пусты, что привело бы к нулевым лимитам при тестировании `@stCostKey`.

**Анализ цепей по наполненности данных:**

| Параметр | Цепь 5 |
|----------|--------|
| Название | «Газпром, 2022-го года, полугодие» |
| ИПГ в цепи | 3 штуки |
| `ipgPnLim` | **8 510 строк** |
| `ipgUtPlPnLmMn` | **9 708 строк** |

**Особенности цепи 5:**
- 3 ИПГ с двумя точками перехода в году: `(дата1_end, дата2_str)` и `(дата3_end, дата4_str)`.
- Дат расчёта: **17** = 2022-01-01 + 12 концов мес. + 4 точки перехода ИПГ.
- Дата 2022-01-01 (начало года) **обязательна**: на неё план/факт = 0, но лимит уже соответствует первой ИПГ цепи; обеспечивает правильный старт графика.
- Два и более ИПГ в рамках одного месяца — **нормально** (каждый имеет свою дату и свой лимит).

**Подтверждение дефекта Б на цепи 5:**

Через `fnIpgChRsltCstUtl_2408`: `SELECT mNum, COUNT(*) ... GROUP BY mNum` — все 3 ИПГ + NULL присутствуют в каждом месяце (4× раздувание, ≈47 520 строк вместо ≈11 880).

**Принятые тестовые параметры:**

```sql
@ipgCh = 5
@MounthEndDate = '2022-09-30'   -- «богатый» месяц: рядом с переходом ИПГ
@ipgStKey = NULL                -- все разделы ИПГ
@stCostKey = NULL               -- без фильтра по затратам
```
