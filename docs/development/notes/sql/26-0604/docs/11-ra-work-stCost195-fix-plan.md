## 1. Постановка проблемы

### 1.1. Каноническая разбивка ОА в отчётах агента

В документах RA (и в нормализованном `ra_summCt`) верхний уровень структуры затрат:

```text
212 (Всего) = 172 (Оборудование) + 187 (Прочие) + 195 (СМР)
```

Узел **182 (Подрядные работы)** — **дочерний** элемент **195 (СМР)**, на одном уровне с материалами (169 и др.), а не замена 195.

### 1.2. Что сделано ошибочно (июнь 2026)

В `01c` / `01d` / `05-fact-stcost-map.md` / `03-design-decisions.md §9` зафиксировано:

```text
ras_work  →  factDocCost @ 182   ← НВЕРНО
```

Семантика колонки Excel **`rainWork` = «СМР»** (`ra_col_map`, id=8) соответствует **195**, не 182.

### 1.3. Влияние на `_2606`

| `@stCostKey` | Legacy `fnStCostRa` (читает `ra_summCt`) | `fnStCostRa_2606` (читает `factDocCost`) сейчас |
|--------------|------------------------------------------|--------------------------------------------------|
| **195** | Сумма СМР (если есть Ct@195) | СМР (через Ct или rollup с ошибочного 182) |
| **182** | Обычно **0** (нет Ct@182 на верхнем уровне) | **Вся СМР** из `ras_work` |
| **169** (материалы под 195) | Только если есть Ct@169 | Почти всегда **0** |

**Регрессия (FishEye dev, 2026-06-13):** **8 674** RA, где `_2606(182) > 0` и `legacy(182) = 0`.

**Следствие для пользователя:** при срезе `@stCostKey=182` вся СМР ошибочно попадает в «подрядные работы»; детализация материалов внутри 195 обнуляется — не соответствует действительности и legacy-поведению.

**Лимиты (`ipgPnLim`) не затронуты** — там уже 172+187+195; трёхчленный тест `07i` по `agLim` проходит.

## 3. Целевая модель данных

### 3.1. Плоские поля RA / RA_change → `factDocCost`

| Плоское поле | stCost | Бизнес-имя | Колонка Excel (Type 5) |
|--------------|--------|------------|-------------------------|
| `*_total` | **212** | Всего с НДС | `rainTtl` |
| `*_work` | **195** | **СМР** | `rainWork` |
| `*_equip` | **172** | Оборудование | `rainEquip` |
| `*_others` | **187** | Прочие | `rainOthers` |

**182** в `factDocCost` для RA появляется **только** из детализирующих строк `ra_summCt` / `ra_change_summCt` (если есть отдельная строка @182), **не** из `ras_work`.

### 3.2. Инварианты приёмки

```text
Факт RA (верхний уровень):
  212 = 172 + 187 + 195

Лимиты ipgPnLim (уже верно):
  212 = 172 + 187 + 195

Плоские поля (как сейчас):
  ras_total = ras_work + ras_equip + ras_others
```

### 3.3. Политика B+F (уточнение Решения 10)

- Плоские поля → `factDocCost` с ключами **212/195/172/187**.
- Шаг 4 `01d`: дополнение из `ra_summCt` — **без изменений**, но при конфликте 195 (flat) vs 195 (Ct) — `NOT EXISTS` / приоритет **не дублировать**.
- Строки Ct с **182, 169, …** под 195 — сохраняются; **не** удалять при миграции.

### 4.3. `01d1_FIX_factDocCost_ra_work_stCost.sql` (новый файл)

**Назначение:** одноразовая коррекция уже залитых данных. Идемпотентен.

**Предусловия:** `01c` с исправленным маппингом (или триггеры DISABLE на время миграции).

**Шаги скрипта:**

1. **Отчёт «до»** — счётчики как в §2.1.
2. **Случай A — есть 182 и 195, суммы равны (±0.01):** `DELETE` строк @182.
3. **Случай B — есть только 182:** `UPDATE fdcoStCost SET 195 WHERE 182` (RaSumm/RaChangeSumm).
4. **Случай C — 182 и 195, суммы различаются:** выгрузить в `#conflicts` (TOP N для ручного разбора); по умолчанию **не** менять автоматически.
5. **Случай D — проверка:** не осталось `RaSumm`/`RaChangeSumm` @182, порождённых из `*_work` *(опционально: JOIN к ra_summ где work совпадает с fdcoSumm)*.
6. **Отчёт «после»** + gate: `regression_182` (см. §5.3) = **0**.

**Откат `01d1`:** только из бэкапа БД или реверс-скрипт с сохранённым `#backup_fdco_182` (рекомендуется SELECT INTO в начале `01d1`).

**Правка `01d_BACKFILL_factDoc.sql`:** синхронно заменить 182→195 в шаге 3 (для повторных установок пакета с нуля).

### 4.4. `07j_COMPARE_stCost_fact_additive_chain5.sql` (новый)

**Назначение:** самосверка **факта** по `@stCostKey` (в отличие от `07i` — лимиты).

**Параметры:** `@ipgCh=5`, `@stIpg` (61 → 46 → NULL), `@epsilon=0.01`.

**Блок LIM** *(опционально, дубль 07i):* `agLim@212 = agLim@172+agLim@187+agLim@195`.

**Блок FACT** *(основной):*

```sql
-- По fnMasteringCstAgPnSh_2606, агрегат по (cstAgPn, dAll):
P212 = SUM(agMasteringPres) @stCostKey=212
P195 = ... @195
P172, P187 — аналогично

PASS_fact: |P212 - (P172+P187+P195)| < ε
           (и то же для agMasteringAccp)
```

**Блок LEGACY parity** *(выборочно, top-N контрактов):*

```sql
|fnStCostRa(ra,195,3) - fnStCostRa_2606(ra,195,3)| < ε
COUNT WHERE fnStCostRa_2606(ra,182,3)>0 AND fnStCostRa(ra,182,3)=0 → 0
```

**Критерий PASS этапа:** stIpg=61 и 46 — `FAIL_fact=0`, `regression_182=0`.

### 4.5. Обновление `07b_VERIFY_fnStCost_2606.sql`

- Тест F: пары `(RA, @StCostKey)` для плоского work — **`@StCostKey=195`**, не 182.
- Явно **исключить** из критерия F пары `(RA, 182)`, где 182 есть только из ошибочного flat (до `01d1`).
- Добавить gate `G-stCost195`: на выборке 100 RA с `ras_work>0`: `fnStCostRa_2606(195) ≈ ras_work`.

## 6. Excel / Java — предотвращение повторения ошибки

> Детали также внесены в документы Type 5 / audit-log (см. §6.4).

### 6.1. Текущее состояние пайплайна

```text
Excel → ra_stg_ra (rainWork = «СМР»)
     → AllAgentsReconcileService.evolveRaSums()
     → INSERT ags.ra_summ (ras_work = staging.work)
     → trgRaSumm_syncFactDoc
     → factDocCost  ← здесь ошибка 182 (до исправления 01c)
```

**Java-код импорта** (`AllAgentsReconcileService`) пишет **корректные** плоские поля; **`ra_summCt` не заполняет**.

### 6.2. Обязательные меры (после исправления 01c)

| # | Мера | Где |
|---|------|-----|
| E1 | Исправленный триггер `01c` — **единственный** путь flat → `factDocCost` | SQL |
| E2 | Post-apply проверка: новые `ra_summ` не создают `factDocCost@182` из work | `type5-post-apply-ra-sanity.sql` §D |
| E3 | В аудит-логе при `RA_NEW_SUMS` / `RA_SUM_MISMATCH` явно: **work = СМР = stCost 195** | `audit-log-vba-to-java-mapping.md` |
| E4 | Чеклист ручной проверки Type 5: пункт про `factDocCost` | `type5-manual-verification-report-template.md` |
| E5 | Комментарий в `ra_col_map` seed / MS_Description: `rainWork` → семантика 195 | Liquibase `2026-03-20-ra-col-map-seed` *(опционально MS_Description)* |

### 6.3. Рекомендуемые меры (этап 2, не блокер SQL-фикса)

| # | Мера | Где |
|---|------|-----|
| E6 | После `evolveRaSums`: синхронизация `ra_summCt` (212/195/172/187) из flat | Java `AllAgentsReconcileService` |
| E7 | Юнит/интеграционный тест: reconcile → `factDocCost` содержит 195, не 182 | `femsq-web` tests |
| E8 | Валидация staging: `ttl = work+equip+others` + предупреждение при отрицательных компонентах | уже частично в reconcile |

### 6.4. SQL gate для Type 5 (§D в `type5-post-apply-ra-sanity.sql`)

После apply с `@baseline_max_ra_key`:

```sql
-- D1: новые ra_summ с work не должны давать factDocCost@182
-- D2: должны давать factDocCost@195 (если work <> 0)
```
