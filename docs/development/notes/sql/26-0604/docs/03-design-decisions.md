## Решение 1: Источник @stNet для DAG-обхода — `ipgCh.ipgcStNetIpg`

**Проблема:** В плане изначально предполагалось использовать `ipg.ipgStRlSh` как источник схемы для `fnStDownAll`. Оба поля дают значение 2 для цепи 15, но семантически разные.

**Принято:** использовать `ipgCh.ipgcStNetIpg`.

**Обоснование:**
- Это атрибут **цепи**, а не конкретной ИПГ — точнее отражает «схема для данной цепи»
- Все существующие функции (`fnMasteringStIpgStCost`, `fnStCostRsStIpg`) берут `@stNet` именно из `ipgCh.ipgcStNetIpg`
- В `fnMasteringStIpgStCost` автор сам оставил комментарий: *«здесь чудно, не может быть структура одна и навсегда!!! Нужно как-то выводить структуру из цепи, видимо по поздней инвестпрограмме»* — и решил через `ipgCh.ipgcStNetIpg`

**Реализация:**
```sql
-- В fnIpgChRsltCstUtl2_2606: JOIN на ipgCh один раз при входе
CROSS JOIN ags.ipgCh ch WHERE ch.ipgcKey = @ipgChKey
-- Затем в фильтре:
AND sp.ipgspSt IN (
    SELECT @ipgStKey UNION SELECT f.strChild
    FROM ags.fnStDownAll(ch.ipgcStNetIpg, @ipgStKey) f
)
```

## Решение 2: fnStDownAll + UNION, не fnSt напрямую

**Проблема:** `fnSt(@stNet, @root)` включает корень, `fnStDownAll` — нет. Для `@ipgStKey=21` (листовой узел) `fnStDownAll(2,21)` возвращает 0 строк.

**Принято:** использовать `fnStDownAll(@stNet, @ipgStKey)` + явный `UNION SELECT @ipgStKey` для корня.

**Обоснование:**
- Это устоявшийся паттерн в кодовой базе (подтверждён в `fnMasteringStIpgStCost` и `fnStCostRsStIpg`)
- `fnSt` как инструмент существует, но в таких сценариях не используется
- Семантически чище: «всё поддерево от @ipgStKey включительно»

## Решение 3: Не использовать fnIpgStNum и ipgSt-семейство

**Проблема:** В предыдущих расчётах использовалась `fnIpgStNumCstapGr` / `fnIpgStNumCstap` / `fnIpgStNum`, работающие с устаревшим семейством таблиц `ipgSt` (созданы 2021-05-31, изолированы от `ipgPn`).

**Принято:** не использовать `fnIpgStNum` и производные в `_2606`.

**Обоснование:**
- Семейство `ipgSt/ipgStRl/ipgStRlSh/ipgStRlShP` не имеет FK-связей с `ipgPn`, `ipg`, `ipgCh`, `ipgStPn`
- Исторически это был первый вариант (2021) структуры ИПГ, заменённый семейством `st` в 2022
- Активные функции в кодовой базе (`fnMasteringStIpgStCost`, `spIpgCh_StPn_Rslt` и др.) используют только `st`-семейство

## Решение 4: ipgChRlV — новая таблица для сроков актуальности

**Проблема (Дефект Б):** `ipg.ipgStr/ipgEnd` — глобальный атрибут ИПГ, а не атрибут связи «цепь → ИПГ». Это приводит к тому, что в месяцах 7–12 одновременно присутствуют ipg=25 и ipg=26.

**Принято:** таблица `ags.ipgChRlV` хранит **только** вводимые атрибуты связи; конец периода **не сохраняется** и **не обновляется** вручную — вычисляется автоматически.

| Колонка | Тип | Смысл |
|---------|-----|-------|
| `ipgcrvKey` | int IDENTITY PK | Суррогатный ключ |
| `ipgcrvChain` | int NOT NULL FK→ipgCh | Цепь ИПГ |
| `ipgcrvIpg` | int NOT NULL FK→ipg | Инвестпрограмма |
| `ipgcrvStr` | date NOT NULL | Дата начала актуальности (**единственная дата, вводимая вручную**) |
| `ipgcrvUtPlGr` | int NULL FK→ipgUtPlGr | Группа планов |
| `ipgcrvEnd` | date NULL **computed** | `ags.fnIpgChRlVEnd(chain, str)` → `MIN(next.str) − 1 день`; NULL = текущая ИПГ |

**Реализация `ipgcrvEnd` — скалярная функция `ags.fnIpgChRlVEnd`:**

Вычисляемый столбец не может напрямую содержать `LEAD` или подзапрос по другим строкам той же таблицы. Обходное решение — скалярный UDF, который вызывается из определения вычисляемого столбца:

```sql
CREATE FUNCTION ags.fnIpgChRlVEnd(@chain int, @str date)
RETURNS date AS
BEGIN
    RETURN DATEADD(day, -1, (
        SELECT MIN(t.ipgcrvStr)
        FROM ags.ipgChRlV t
        WHERE t.ipgcrvChain = @chain AND t.ipgcrvStr > @str
    ))
END

ALTER TABLE ags.ipgChRlV
    ADD ipgcrvEnd AS (ags.fnIpgChRlVEnd(ipgcrvChain, ipgcrvStr));
```

**Преимущества перед представлением:** `ipgcrvEnd` доступен непосредственно при обращении к `ags.ipgChRlV` — все функции `_2606` читают базовую таблицу, не требуя отдельного объекта-обёртки. Производительность не является проблемой для справочной таблицы малого размера.

**Запрещено:** `INSERT`/`UPDATE` значения `ipgcrvEnd` — столбец вычисляемый, запись в него невозможна.

**Данные для цепи 15 (первый этап):**

| ipgcrvChain | ipgcrvIpg | ipgcrvStr | ipgcrvEnd (вычисл.) | ipgcrvUtPlGr |
|-------------|-----------|-----------|----------------------|--------------|
| 15 | 25 | 2024-11-28 | 2025-07-15 | 17 |
| 15 | 26 | 2025-07-16 | NULL | NULL |

**Почему `ipgcrvUtPlGr = NULL` для ipg=26:**  
В `ags.ipgUtPlGr` нет записи для ipg=26 (последняя — key=17 для ipg=25). «Письмо Д644» не имеет выделенной группы планов на момент реализации.

## Решение 5: Параметр @ipgStKey int, не @ipgSt nvarchar

**Принято:** `@ipgStKey int` (числовой ключ `stIpg.stiKey`).

**Обоснование:** `@ipgSt nvarchar` в `_2605` содержит коды типов затрат (`importIpgSt_26-0320.cst_type`), а не структуру ИПГ. Это разные измерения; `_2606` решает другую задачу.

## Решение 7 (пересмотрено): базовый стек — `fnMasteringStIpgStCost`, не `fnIpgChRsltCstUtl_2408`

**История решения:**

_Первоначально (Q3 = нет @stCostKey)_ предполагалось оставить `fnIpgChRsltCstUtl_2408` как базу и добавить только `@ipgStKey`.

_Пересмотрено_ в ходе анализа (2026-06-04) после исследования `fnMasteringStIpgStCost`, `fnStCostRsCstAgPn`, `fnStCostRsIpgPn`, `ipgPnLim`, `ipgUtPlPnLmMn`.

**Принято:** `spMstrg_2606` строится на стеке `fnMasteringStIpgStCost`, оба параметра `@ipgStKey` и `@stCostKey` включены с nullable-семантикой (NULL = без фильтра).

**Обоснование:**

1. `spMstrg_2605` является **частным случаем** `fnMasteringStIpgStCost` при `@ipgStKey=NULL, @stCostKey=NULL` — это «взять всё, не фильтровать». Разделять на два прохода значит дважды писать по сути один и тот же код.

2. `fnMasteringStIpgStCost` уже содержит **встроенный Дефект А** (комментарий автора: *«здесь чудно, не может быть структура одна и навсегда!!!»*) — там же `max(c.ipgcStNetIpg)`, т.е. всегда берётся последняя ИПГ. Исправление этого требует версионированной копии функций в любом случае.

3. Для цепи 15 `ipgPnLim` и `ipgUtPlPnLmMn` пусты, однако данные планируется наполнить. Для сохранения совместимости в `fnStCostRsIpgPn_2606` вводится **расширенный fallback** к `ipgUtPlP.iuplpLim` (суммарный лимит), что позволяет функции работать ДО наполнения `ipgPnLim`.

**Реализация: новые DB-объекты (9 шт.)**

| Объект | Тип | Назначение |
|--------|-----|-----------|
| `ags.ipgChRlV` | TABLE | Хранит `ipgcrvStr` (начало в цепи) |
| `ags.vIpgChRlV` | VIEW | + вычисляемый `ipgcrvEnd` |
| `ags.fnIpgChDatsV` | IF | Генератор дат цепи с точкой разрыва |
| `ags.fnStCostRsIpgPn_2606` | TF | Фикс Деф.Б + fallback к `ipgUtPlP.iuplpLim` |
| `ags.fnStCostRsCstAgPn_2606` | TF | Использует `_2606`-подфункцию |
| `ags.fnMasteringCstAgPnSh_2606` | TF | Использует `_2606`-подфункции |
| `ags.fnMasteringStIpgStCost_2606` | IF | Фикс Деф.А (stNet per-IPG из `ipgChRlV`), `@ipgStKey`/`@stCostKey` nullable |
| `ags.fnIpgChRsltCstUtl2_2606` | TF | Обёртка: добавляет метаданные, совместима по формату с `_2605` |
| `ags.fnIpgChRsltCstUtlPercentBrn_2606` | IF | Процентный показатель |
| `ags.spMstrg_2606` | P | Хранимая процедура |

## Целевая сигнатура `spMstrg_2606`

```sql
EXEC ags.spMstrg_2606
    @ipgCh          int,           -- цепь ИПГ
    @MounthEndDate  date,          -- последний день расчётного месяца
    @ipgStKey       int    = NULL, -- узел stIpg (NULL = без фильтра по разделу ИПГ)
    @stCostKey      int    = NULL, -- узел stCost (NULL = без фильтра по статье затрат)
    @saveToTables   bit    = 0     -- 0: SELECT×7, 1: INSERT в spMstrg_2606_ResultSet1..7
```

**Инварианты:**
- `spMstrg_2606(15, '2025-07-31', NULL, NULL)` ≡ `spMstrg_2605(15, '2025-07-31', NULL)` (с поправкой на исправленные дефекты)
- `@stCostKey` работает для всех цепей немедленно; для цепи 15 фильтрация станет точной после наполнения `ipgPnLim`
- Параметры независимы: можно фильтровать по одному, по обоим или не фильтровать совсем

### Семантика `@ipgStKey` (фильтр строек, не декомпозиции)

**Принято (2026-06-26):** `@ipgStKey` задаёт **подмножество строек** (`cstAgPn`) в цепи, а **не** срез пунктов ИП или помесячной разбивки UtPl внутри стройки.

| Уровень | Что фильтрует `@ipgStKey` | Что **не** фильтрует |
|---------|---------------------------|----------------------|
| Отбор строек | Стройка попадает в расчёт, если **хотя бы один** её `ipgpPn` связан с узлом `stIpg` (или потомком через `fnStDownAll`) в `ipgStPn` | — |
| Внутри стройки | — | Все пункты ИП цепи (`ipgChRlV` × `ipgPn`), декомпозиция `ipgUtPlPnLmMn` по месяцам и stCost, смены ИП 6/8/11 |

После отбора `fnMasteringStIpgStCost_2606` вызывает `fnMasteringCstAgPnSh_2606` **на всю стройку** (`@cstAgPn`), со всеми пунктами ИП и UtPl в тестовых группах.

**Следствия:**

1. Если стройка прошла фильтр при `@ipgStKey = A` и при `@ipgStKey = B`, набор строк по ней в `fn2` / `spMstrg_2606` **идентичен** (проверено dev: `cstAgPn = 2102`, `stIpg` 42 vs 61, `EXCEPT` = 0).
2. Разные `@ipgStKey` меняют только **состав строек** в полном прогоне, не «полноту» одной стройки.
3. **Не путать:** узлы `stIpg` (дерево разделов ИПГ для фильтра отчёта) и **инвестпрограммы** в цепи (`ipgChRlV`, ИП 6/8/11 …) — разные измерения. Количество привязок `ipgStPn` у стройки к разным `stIpg` **не означает** «частичный» UtPl в отчёте.

**Ручной прогон одной стройки:** `@ipgStKey`, при котором стройка входит в фильтр (напр. 42 или 61 для golden `cstAgPn = 2102`), либо `@ipgStKey = NULL` + `WHERE cstAgPnKey = …` в ResultSet. См. `13-plan-stcost-monthly-acceptance.md` §17, [`glossary.md`](../../../../../project/glossary.md).

*(параметр `@stNet` не нужен: берётся из `ipgCh.ipgcStNetIpg` → `ipgChRlV` per-date внутри функций)*

## Решение 8: Параллельное сосуществование — отдельные таблицы ResultSet и независимые клиенты

**Проблема:** В черновике архитектуры `spMstrg_2606` при `@saveToTables = 1` фигурировали те же таблицы `ags.spMstrg_2408_ResultSet1..7`, что используют `spMstrg_2408`, `spMstrg_2605` и FEMSQ/JasperReports. Запись `_2606` в них **перезаписала бы** данные действующих клиентов и нарушила бы принцип «не трогаем `_2605`».

**Принято:** полная изоляция persistent-артефактов по суффиксу версии.  
**Подтверждено пользователем:** 2026-06-04 (согласовано перед Этапом 1).

| Слой | `_2408` / `_2605` (без изменений) | `_2606` (новое) |
|------|-----------------------------------|-----------------|
| SQL Server, режим FEMSQ | `ags.spMstrg_2408_ResultSet1..7` | **`ags.spMstrg_2606_ResultSet1..7`** (CREATE в пакете) |
| Процедура `@saveToTables=1` | `spMstrg_2605` → TRUNCATE/INSERT в `*_2408_ResultSet*` | `spMstrg_2606` → только в `*_2606_ResultSet*` |
| MS Access, локальные таблицы | `ags_fnIpgChRsltCstUtlPercentBrn_2408_tbl_part_*`, `*Rep01_2408_tbl`, … | **Отдельные имена с `_2606`** (создать в Access при переключении; инструкция в deploy-документе) |
| FEMSQ / JasperReports | `execute_spMstrg_2605.sh` + JRXML на `*_2408_ResultSet*` | **`execute_spMstrg_2606.sh`** + смена источника на `*_2606_ResultSet*` (после SQL-приёмки) |

**Структура таблиц:** `spMstrg_2606_ResultSet1..7` — **та же схема столбцов**, что у `spMstrg_2408_ResultSet1..7` (процедура `_2606` строится от того же шаблона SELECT, меняются только имена целевых таблиц и вызовы `fn*_2606`).

**Инварианты совместимости:**
- `spMstrg_2605` и `spMstrg_2408*` продолжают работать как до деплоя `_2606`.
- Деплой `_2606` на продуктив **не обязан** сразу переключать Access/FEMSQ — достаточно создать объекты и таблицы; переключение клиентов — отдельный шаг в `db-upgrade-spMstrg-2606.md`.
- Откат `08_ROLLBACK.sql`: DROP `_2606`-объектов **и** DROP `spMstrg_2606_ResultSet1..7`; таблицы `_2408_ResultSet*` не затрагиваются.

**В пакете SQL (добавить):**
- `05b_CREATE_TABLE_spMstrg_2606_ResultSets.sql` — DDL семи таблиц (dev + `MSSQL2012/`).
- `06_CREATE_PROCEDURE_spMstrg_2606.sql` — только `spMstrg_2606_ResultSet*`.
- `07_VERIFY_after.sql` — COUNT из `spMstrg_2606_ResultSet*`, плюс проверка что `spMstrg_2605` по-прежнему пишет в `*_2408_ResultSet*`.

## Решение 9: Двухшаговая модель + суперкласс `factDoc` для всех документов факта (шаг 1)

**Принято:** 2026-06-05 (подтверждено пользователем).

### Двухшаговая модель работ

| | Шаг 1 (этот чат) | Шаг 2 (после приёмки `_2606`) |
|---|---|---|
| **Цель** | Работоспособная `spMstrg_2606` + SQL-пакет для продуктива | Комфортная эксплуатация: клиенты, данные, опциональные улучшения |
| **SQL** | Все объекты `_2606`, `factDoc`/`factDocCost`, триггеры, бэкфилл | Переключение Access/FEMSQ на `*_2606_ResultSet*` |
| **Клиенты** | **Не меняем** VBA, Java, Access | Адаптация по необходимости |
| **Данные** | Ручные правки dev (лимиты цепи 15 и т.п.) — вне пакета или пометка `manual` | Наполнение `ipgPnLim` на продуктиве и др. |
| **Риск** | Минимальный: старый контур изолирован (Решение 8) | Отдельный план работ |

**Инвариант шага 1:** `_2605` / `_2408`, `fnStCostRa`, `ra_summCt` и текущие клиенты работают как до деплоя. Новый контур `_2606` — параллельный, не заменяющий.

### Суперкласс `factDoc` — все типы документов в шаге 1

**Принцип:** на шаге 1 **все** типы документов факта, участвующие в расчёте `_2606`, переводятся на единую схему `factDoc` + `factDocCost`. Частичная реализация (только RA) **не допускается** — теряется однообразие стека и смысл суперкласса.

**Минимальная схема (без изменений от ранее согласованного):**

| Таблица | Назначение |
|---------|------------|
| `ags.factDoc` | `fdKey` PK; `fdDocType` nvarchar; `fdNKey` int — обратная связь с подклассом |
| `ags.factDocCost` | `fdcoFd` → factDoc; `fdcoStCost` → stCost; `fdcoSumm` money *(префикс `fdco` = fact **D**oc **Co**st)* |

**Подклассы и привязки (шаг 1):**

| `fdDocType` | Таблица-подкласс | Колонка FK | Плоские поля → `fdcoStCost` |
|-------------|------------------|------------|------------------------------|
| `RaSumm` | `ra_summ` | `ras_fdKey` | total→212, **work→195**, equip→172, others→187 |
| `RaChangeSumm` | `ra_change_summ` | `racs_fdKey` | total→212, **work→195**, equip→172, others→187 |
| `OgAgFeeP` | `ogAgFeeP` | `oafp_fdKey` | `oafpTotal`→148 |
| `RalpRaAu` | `ralpRaAu` | `ralpra_fdKey` | `ralpraCostAndVat`→150 |
| `PrDocP` | `cn_PrDocP` | `pdp_fdKey` | `costVAT`→205 (ZPTG/ZKTG) или →197 (ZUGH) по `pdtoCode` |
| `CstAgPnMnrl` | `cstAgPnMnrl` | `am_fdKey` | `amSum`→169 |

**Триггеры `AFTER INSERT, UPDATE`:**

- На INSERT: создать строку в `factDoc`, присвоить `*_fdKey` подклассу.
- На INSERT/UPDATE с суммами: синхронизировать `factDocCost`; для VBA — фильтр `UPDATE(column)` (срабатывает только когда плоское поле реально записано).
- В части синхронизации `factDocCost` читать `*_fdKey` из **реальной таблицы** (JOIN), не из `inserted` — иначе одношаговый INSERT Java не получит `fdKey` в той же операции.

**Функции `_2606` читают только `factDocCost`:**

| Старая функция | Новая `_2606` | Источник сумм |
|----------------|---------------|---------------|
| `fnStCostRa` | `fnStCostRa_2606` | `factDocCost` по `ras_fdKey` |
| `fnStCostRaCh` | `fnStCostRaCh_2606` | `factDocCost` по `racs_fdKey` |
| `fnStCostAgFee` | `fnStCostAgFee_2606` | `factDocCost` по `oafp_fdKey` |
| `fnStCostRalp` | `fnStCostRalp_2606` | `factDocCost` по `ralpra_fdKey` |
| `fnStCostPrDoc` | `fnStCostPrDoc_2606` | `factDocCost` по `pdp_fdKey` |
| `fnStCostMnrl` | `fnStCostMnrl_2606` | `factDocCost` по `am_fdKey` |

## Решение 10: Политика хранения `factDocCost` — вариант B+F

**Принято:** 2026-06-05 (подтверждено пользователем после проверки бэкфилла на FishEye dev).

### Контекст

После `01d_BACKFILL_factDoc.sql` проверена согласованность `factDocCost` с источниками `ra_summ` / `ra_summCt` (и аналогично `ra_change_summ` / `ra_change_summCt`). Вопрос: не произошло ли задвоение сумм при объединении плоских полей и Ct.

### Результаты проверки (FishEye dev)

| Проверка | Результат |
|----------|-----------|
| Плоские поля `ra_summ` → `factDocCost` (212/182/172/187) | **0** расхождений значений |
| Строки `ra_summCt` → `factDocCost` | **19 878 / 19 878** совпадают; **0** пропусков |
| Дубль одного `(fdcoFd, fdcoStCost)` из двух источников | **0** (`NOT EXISTS` + `UQ_factDocCost_fd_stcost`) |
| Наивный `SUM(factDocCost)` ≈ 2× `ras_total` | **~23 522** из 27 630 RA — ожидаемо: одновременно итог (212) и детализация (182/172/187) |

**Вывод:** построчного задвоения при переносе нет. «Раздувание» при наивном `SUM` — следствие иерархической модели хранения (итог + части), как в legacy `ra_summCt`.

### Вариант B — хранение

- **`01d` / `01c` не пересматриваем:** в `factDocCost` остаётся объединение плоских полей и дополнительных строк Ct (шаг 4).
- Триггер на `ra_summCt` **не добавляем** (шаг 2).
- Политики «только Ct» или «либо 212, либо 182/172/187» **не применяем**.

### Вариант F — приёмка функций `_2606`

Новые `fnStCost*_2606` **точно повторяют DAG-логику** соответствующих legacy-функций, но читают `factDocCost` вместо `ra_summCt` / плоских полей:

1. Найти «последний» носитель по дате (как в `fnStCostRa` / `fnStCostRaCh`).
2. Точное значение по `@StCostKey` в `factDocCost`.
3. Если NULL — `fnStDownAll` + исключение перекрытых узлов (копия алгоритма legacy).

**Критерий приёмки (тест F):**

```text
|fnStCost*(carrier, @StCostKey, @stNet) - fnStCost*_2606(...)| < 0.01
```

на выборке пар **(носитель, @StCostKey)**, где на **последней** сумме носителя есть строка `ra_summCt` / `ra_change_summCt` с этим `@StCostKey` (не декартово «любой Ct × все ключи 212/182/…»).

**Наивный `SUM(factDocCost)` по документу — не критерий приёмки.**

**Ожидаемое отличие (не регрессия):**
- RA **без** `ra_summCt` целиком, но с плоскими полями — `fnStCostRa` = 0, `fnStCostRa_2606` > 0.
- RA **с** Ct, но для конкретного `@StCostKey` только flat (напр. 182), а в Ct другой ключ (напр. 195) — legacy = 0, `_2606` > 0; такие пары **не входят** в тест F.

### Скрипт приёмки

`07b_VERIFY_fnStCost_2606.sql` (или блок в `07_VERIFY_after.sql`) — выполняется после `03b0` на dev.

## Решение 14: План на датах смены ИП — «ужесточение» `_2605`; интерполяция — отложенная задача

**Дата:** 2026-06-23 | **Статус:** ✅ принято (политика приёмки); 📋 отложено (интерполяция плана и факта по дням)

### Контекст

В стеке `_2605` актуальность ИП определяется по **месяцу**; накопленный план на месяц смены включает **целый** месячный прирост UtPl. В `_2606` актуальность — по **дню** (`ipgChRlV`, `fnIpgChDatsV`), но `fnStCostRsIpgPn_2606` по-прежнему накапливает **полные** месячные `iuplpmLim` в `smmTtl` на датах `ipgcrvEnd` / `ipgcrvStr` (parity с «ужесточением» `_2605`).

Эффект: при смене ИП в середине месяца (например, 26–27.04) исполнитель формально «должен» освоить план за большую долю месяца, чем календарные дни активности ИП. Аналогичный сдвиг возможен и для **факта**, если документы засчитываются по датам периода без пропорции к промежуточному дню.

### Принято (текущая реализация и приёмка)

1. **Сохранить** «ужесточение» из `_2605` в `_2606` — без изменения `03a` / mastering до отдельного решения.
2. Критерии приёмки планов (**К-12c**, **К-12t**): на `ipgcrvEnd` cum(UtPl) **без** пропорции по дню месяца.
3. Термин **«ревизия»** не применять к смене ИП; см. `docs/project/glossary.md`.

### Отложенная задача (объёмная, возможно позже)

Для **корректной** работы на промежуточных днях месяца потребуется:

| Контур | Требование |
|--------|------------|
| **План** | Интерполяция накопленного плана по дням в месяце смены ИП (пропорция активных дней или иная согласованная модель). |
| **Факт** | Определение по **датам из документов факта**, что именно принято/представлено к промежуточному дню (симметричный cutoff с планом). |

Затрагивает: `fnStCostRsIpgPn_2606`, функции mastering факта, возможно модель UtPl и клиентский отчёт. **Не блокер** релиза SQL-пакета `_2606` при текущей политике.

**Связанные этапы чат-плана:** 18.9 (проектирование), реализация — после 18.7.4.

## Решение 15: Sparse-хранение UtPl и CHECK `lim > 0`

**Дата:** 2026-06-26 | **Статус:** ✅ принято (политика данных, fixture, deploy)

### Контекст

1. Таблицы нормализованной разбивки UtPl (`ipgUtPlPnLmMn`, `ipgUtPlPnLmQu`, `ipgUtPlPnLmYe`) изначально снабжены CHECK `iuplp*Lim <> 0`, но ограничения **отключены** (DISABLED / NOT TRUSTED).

| Утверждение | Деталь |
|-------------|--------|
| Хранить только `lim > 0` | Строка = «в этом периоде **есть** плановое освоение» |
| Нет плана в месяце | **Нет строки** в `ipgUtPlPnLmMn` (не `iuplpmLim = 0`) |

#### Fixture и dev-приёмка

#### Миграция и CHECK

| Шаг | Содержание |
|-----|------------|
| **09a** | Аудит `lim <= 0` (отчёт) |
