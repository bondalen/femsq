**Дата:** 2026-06-24  
**Статус:** **принято** → `03-design-decisions.md` **Решение 16**; реализация — чат-план **этап 19**  
**Контекст:** пакет `26-0604`, `spMstrg_2606`, модель «группа + внелимит»  
**Связанные документы:** `03-design-decisions.md`, `04-computation-map.md`, `06-sp-recordsets-and-acceptance.md`, `13-plan-stcost-monthly-acceptance.md` §17

## 1. Постановка проблемы

При прогоне `spMstrg_2606` с `@ipgStKey` (например, `42` или `61` для golden-стройки `cstAgPn = 2102`) в рекордсетах остаётся **много строек**, хотя в `ipgStPn` под узлом — одна стройка.

Это не ошибка fixture, а **расхождение двух слоёв**:

| Слой | Поведение сегодня | Ожидаемая бизнес-модель |
|------|-------------------|-------------------------|
| **Освоение / план** (`fnMasteringStIpgStCost_2606`) | Только стройки из **поддерева `ipgStPn`** | То же + **внелимитные** по правилам узла |
| **Рекордсет** (`fn2_2606` → `PercentBrn` → `spMstrg`) | **Все стройки цепи** (~680 на цепи 5; наследие `fn_2408`) | Только «в группе» ± внелимит по `stiLim` |

**Наблюдение dev (цепь 5, `@stCostKey = 212`, `@ipgStKey = 42`):**

| Метрика | Значение |
|---------|----------|
| Строек в `ipgStPn` под узлом 42 | 1 (`2102`) |
| Строк `fnMasteringStIpgStCost` по 2102 | есть |
| Разных `cstapKey` в `PercentBrn` | ~829 |
| Строк `PercentBrn` только по `cstapKey = 2102` | **64** (= **16** `dateRslt` × 4 уровня GROUPING SETS; legacy календарь без 01.01 — см. **Решение 17**) |

**Вопрос:** должна ли процедура типа `_2606` возвращать данные по стройкам **вне** выбранного раздела ИПГ, или ограничиться **группой** (и при необходимости — внелимитными по типу)?

## 2. Бизнес-модель (целевая семантика)

Пример иерархии разделов ИПГ:

```
1.     «всего»
1.1.   «капитальное строительство»
1.1.1. «геологоразведка»
1.2.   «ПИР»
1.2.1. «ПИР для геолоразведки»
```

| Узел | Состав отчёта |
|------|----------------|
| **1. «всего»** | Всё в группе + любая стройка **вне** группы, где есть **лимит** или **принято** что-то вне лимита |
| **1.1. КС** | Группа + внелимитные стройки вида `51-2001234` (признак **200** = строительство, не ПИР) |
| **1.2. ПИР** | Группа + внелимитные с типом **`1`** (5-я литера) |
| **1.1.1, 1.2.1** (листья) | **Только** стройки группы (`ipgStPn`), без внелимитных |

**Тип стройки — 5-я литера кода САК** (`cstapIpgPnN`):

| Код | 5-я литера | Смысл | Пример |
|-----|------------|-------|--------|
| ПИР | `1` | проектно-изыскательские работы | `051-1001234` |
| КС | `2` | стройка | `051-2001234` |
| Бурение | `3` | бурение | `051-3001234` |

Функция: `SUBSTRING(cstapIpgPnN, 5, 1)` → `ags.fnCstAgPnTypeChar`.

## 3. Метаданные в БД: `stiLim` и `ipgStLm`

### 3.1. Таблица `ags.stIpg`

| Колонка | Назначение |
|---------|------------|
| `stiKey` | Ключ узла (= `st.stKey`) |
| `stiLim` | FK → `ipgStLm` (nullable); правило внелимитных для узла |
| `stiType`, `stiNote` | Служебные |

### 3.2. Справочник `ags.ipgStLm`

| `ipgsplKey` | `ipgsplName` |
|-------------|--------------|
| 1 | безлимитные **все** |
| 2 | безлимитные **стройки** (КС, код `2xx`) |
| 3 | безлимитные **ПИР** (код `1xx`) |

### 3.3. Примеры узлов (FishEye, `stNet = 2`)

| `stiKey` | Имя | `stiLim` | Смысл |
|----------|-----|----------|-------|
| 1 | Вся инвестпрограмма | 1 | группа + все внелимитные с активностью |
| 2 | План капитального строительства | 1 | то же (широкий узел) |
| 51 | План капитального строительства без ПИР | 2 | группа + внелимитные КС |
| 45 | ПИР и РД | 3 | группа + внелимитные ПИР |
| 42, 61 | Геологоразведка (раздел / виды) | NULL | **только** группа |
| остальные ~78 узлов | — | NULL | только группа |

**Факт:** `stiLim` заполняется при вставке узла (триггер `stIpg_insert`), но **ни одна функция расчёта** в схеме `ags` не читает `stiLim` / `ipgStLm` (проверка `sys.sql_modules`, 2026-06-24).

## 4. Историческая эволюция (почему не довели)

```
fn_2408          → все контракты цепи в рекордсете
fn2_2605         → фильтр @ipgSt nvarchar через importIpgSt_26-0320 (ручной список cst → раздел)
2022 stIpg       → DAG + stiLim + ipgStLm (задумка «группа + внелимит»)
fnMastering*     → только ipgStPn ∩ поддерево; stiLim не подключён
_2606            → @ipgStKey int в mastering; fn2 без фильтра по разделу
```

### 4.1. `spMstrg_2605` / `fn2_2605`

Фильтр `@ipgSt = N'12ОПР'` в конце `fn2_2605`:

```sql
WHERE (@ipgSt IS NULL OR EXISTS (
    SELECT 1 FROM ags.[importIpgSt_26-0320] s
    WHERE s.cst = asd.cstAgPnCode AND s.cst_type = @ipgSt
))
```

RS1 при `@ipgSt = NULL`: ~12 693 строк; при `@ipgSt = '12ОПР'`: ~604 строки.

### 4.2. Стек mastering (2022)

`fnMasteringStIpgStCost` → `fnMasteringCstAgPnSh` → `fnMasteringCstAgPn` → `fnStCostRsCstAgPn(@ipgRoot)`.

`fnStCostRsCstAgPn`: если стройка **не** в поддереве `@ipgRoot` (`fnStUpAll` + `ipgStPn`), план/лимит **пустые** — но это не отсекает строку из `fn2`.

### 4.3. `fn2_2606`

- `@ipgStKey` влияет на CTE `mastering` (`fnMasteringStIpgStCost_2606`).
- CTE `ipgChContracts` строится по **всей цепи** (RA, ogAgFee, `ipgPn`, …) — как `fn_2408`.
- **Нет** аналога фильтра `importIpgSt` из `_2605`.

**Регрессия:** при `@ipgStKey ≠ NULL` фильтрация рекордсета **слабее**, чем в `_2605`.

## 5. Два режима ответа на вопрос «чужие стройки в RS»

### A. `@ipgStKey` задан (отчёт по разделу)

Должен включать:

1. **IN_GROUP** — стройки с `ipgStPn` в поддереве узла (`fnStDownAll` + корень).
2. **OUT_GROUP** — по `stiLim` узла (если не NULL):
   - стройки цепи **вне** IN_GROUP;
   - с активностью: лимит > 0 **или** есть принятый/представленный факт (уточнить предикат при реализации);
   - с фильтром кода: `1` — все, `2` — `2xx`, `3` — `1xx`.

### B. `@ipgStKey = NULL` (вся цепь)

Полный универсум цепи — текущее поведение (~14 447 строк RS1 для цепи 5).

## 6. Предложения (по приоритету)

### 6.1. Документация — «Решение 18» (без кода)

Зафиксировать в `03-design-decisions.md`:

- семантику `stiLim` + `ipgStLm`;
- два универсума (mastering vs RS);
- регрессию относительно `_2605`;
- что `@ipgStKey = 42` **не** означает «только одна стройка в RS».

**Статус:** частично отражено в § `@ipgStKey` (2026-06-26); расширить блоком про `stiLim` и RS.

### 6.2. Реализовать `fnIpgChContractsForStIpg_2606` (рекомендуемый путь)

Единая функция «кто входит в отчёт при `@ipgStKey`»:

```text
UNION(
  IN_GROUP:  DISTINCT ipgpCstAgPn
             FROM ipgChRl_2606 × ipgPn × ipgStPn
             WHERE ipgspSt ∈ fnStDownAll(@stNet, @ipgStKey) ∪ {@ipgStKey}

  OUT_GROUP: IF stiLim(@ipgStKey) IS NOT NULL:
             cstAgPn цепи \ IN_GROUP
             WHERE (has_lim OR has_fact)  -- уточнить предикат
               AND code_match(stiLim)     -- 1=all, 2=2xx, 3=1xx
)
```

Подключить в CTE `ipgChContracts` (и при необходимости `nullIpgBase`) в `fn2_2606`.

**Эффект:** `@ipgStKey = 42` → RS только по стройкам группы (для dev — 2102), без 679 «чужих» строк.

### 6.3. Переходный компромисс — parity с `_2605` (минимальный diff)

Пока `stiLim` не реализован — финальный фильтр в `fn2_2606`:

```sql
WHERE @ipgStKey IS NULL
   OR cstAgPnKey IN (SELECT ipgpCstAgPn FROM <IN_GROUP по ipgStPn>)
```

Убирает лишние стройки в RS, **без** внелимитных.

### 6.4. Расширение API `spMstrg_2606`

```sql
@contractScope tinyint = 0
-- 0 = как сейчас (вся цепь в fn2)
-- 1 = только IN_GROUP
-- 2 = IN_GROUP + OUT_GROUP по stiLim (полная модель)
```

Дополнительно для dev: опциональный `@cstAgPnKey` (spot-check одной стройки).

### 6.5. Что не делать

| Подход | Причина отказа |
|--------|----------------|
| Новая цепь с 3 ИП и одной стройкой | Не решает общую модель; огромный fixture |
| «Экзотический» `stIpg` только с одной привязкой | Уже есть (42, 61); RS всё равно полная цепь без п. 6.2 |
| Вернуть `importIpgSt_26-0320` | Временный костыль; дублирует `stiLim` |

### 6.6. Приёмка после реализации п. 6.2

Тест-кейсы на цепи 5:

| `@ipgStKey` | `stiLim` | Ожидание |
|-------------|----------|----------|
| 1 | 1 | IN_GROUP + все внелимитные с активностью |
| 51 | 2 | IN_GROUP + КС (`2xx`) вне группы |
| 45 | 3 | IN_GROUP + ПИР (`1xx`) вне группы |
| 42 | NULL | **только** IN_GROUP (2102) |
| NULL | — | вся цепь |

Сверка: `COUNT(RS1)`, ключевые поля на `dateRslt`, `SUM(ag_presented)` vs `fnMasteringStIpgStCost` на 17 датах `fnIpgChDats_2606`.

## 8. Открытые вопросы (этап 19)

1. **Предикат активности** для OUT_GROUP: только `lim > 0`, или `accepted > 0`, или `limNot`/`limOver`, или объединение?

## 10. Принятое решение (2026-06-24)

**Хранение:** одна таблица `ags.stIpgOutLimPn_2606` — прямая связь **узел `stIpg` → много типов** (`char(1)`). Без профилей `ipgStLm`, без строк `"2,3"` в колонке.

**Seed (минимум):**

| `stiKey` | `siolpCstType` |
|----------|----------------|
| 1 | `1`, `2`, `3` |
| 2 | `1`, `2`, `3` *(при равенстве правилу узла 1)* |
| 51 | `2`, `3` |
| 45 | `1` |
| 42, 61, листья | *(нет строк)* |

**Расчёт:** `fnIpgChContractsForStIpg_2606` → фильтр `ipgChContracts` в `fn2_2606`. `@ipgStKey = NULL` — вся цепь.
