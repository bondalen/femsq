## 1. Назначение

Подтвердить на **dev** (Docker `femsq-mssql`), что стек `_2606` корректно обрабатывает **помесячные плановые показатели** уточнённого плана (`ipgUtPlPnLmMn`) с разбивкой по элементам структуры затрат **stCost** и в целом.

На **продуктиве** разбивка UtPl по stCost — отдельная бизнес-задача (наполнение данными организациями, длительный срок). **Процесс prod-деплоя не меняется:** fixture и скрипты фазы 3 **не входят** в `MSSQL2012/` и на флеш-носитель.

Успешная dev-приёмка по планам подтверждает **логику стека**; отсутствие UtPl@172/187/195 на prod до наполнения данных — **не блокер** релиза SQL-пакета.

## 3. Главный критерий качества плана (К-12)

Для каждой стройки с помесячной разбивкой **накопленный план на дату** должен **строго совпадать с лимитом** `ipgPn` по **каждому** stCost отдельно.

| stCost | Смысл | Эталон лимита | Поле стека (`fnMasteringCstAgPnSh_2606`) |
|--------|-------|---------------|----------------------------------------|
| 212 | Итог | `ipgpSmTtl × 1 000 000` | `agSmmTtl` @212 |
| 195 | СМР (work) | `ipgpSmWrk × 1 000 000` | `agSmmTtl` @195 |
| 172 | Оборудование | `ipgpSmEqu × 1 000 000` | `agSmmTtl` @172 |
| 187 | Прочие | `ipgpSmOth × 1 000 000` | `agSmmTtl` @187 |

Эквивалентно: `ipgPnLim.ipgplLim × 1 000 000` при наличии строки на соответствующем stCost (на цепи 5 согласовано с `ipgpSm*`).

**Пример:** лимит @195 = 100 ₽ → сумма всех месяцев `iuplpmLim@195` = 100 ₽ и `agSmmTtl@195` на `2022-12-31` = 100 ₽.

В PercentBrn накопленный план отображается как `ag_Pl` / `ag_PlAccum` (источник — тот же контур `smm` / `smmTtl`).

## 4. Критерий согласованности частей (К-13)

На той же дате и той же стройке (с UtPlMn):

```
agSmmTtl@212  ≈  agSmmTtl@172 + agSmmTtl@187 + agSmmTtl@195
```

Допуск: `ε = 0,01` (money).

К-13 **не заменяет** К-12: сначала каждая статья сходится с лимитом, затем итог сходится с суммой частей.

## 5. Двухуровневая проверка (скрипт `07m_plan_limit_conformance_chain5.sql`)

### Фаза 0 — данные `ipgUtPlPnLmMn` (секунды)

По каждому `ipgPn` с UtPlMn и каждому stCost ∈ {212, 195, 172, 187}:

```
sum(iuplpmLim по всем месяцам)  ≈  ipgplLim@stCost
```

При FAIL фаза 1 не запускается.

### Фаза 1 — стек `_2606` (4 прохода TVF)

По каждому `ipgPn` с UtPlMn, **последний** `dAll ≤ @MounthEndDate`, где план ненулевой:

```
COALESCE(agSmmTtl, inSmmTtl, drSmmTtl)  ≈  cum(UtPlMn) × 1 000 000
```

где `cum(UtPlMn)` — накопленная сумма `iuplpmLim` по `iuplpmMn` (любой префикс месяцев; на последней дате освоения стека).

На `2022-12-31` у большинства пн план в стеке **нулевой** (нет строки `dAll`); это не FAIL — берётся последняя дата с планом. `SKIP` — нет ни одной даты с планом по stCost.

**PERF (К-12):** wall-clock **≤ 480 с** на полную цепь (4 stCost × NULL). Факт **~300 с** (2026-06-17).

## 6. Аддитивность плана (скрипт `07m_plan_additive_chain5.sql`)

По образцу `07i_COMPARE_stCost_additive_chain5.sql`, поле `COALESCE(ag/in/dr SmmTtl)`, последний `dAll ≤ @dt`, уровень **ipgPn**:

```
plan@212  ≈  plan@172 + plan@187 + plan@195
```

Допуск: `ε = 5,00` (money) — накопление ×1e6 после split fixture по весам `ipgPnLim` (макс. наблюдённый |diff| ≈ 2,06).

**PERF (К-13):** wall-clock **≤ 480 с** на полную цепь. Факт **~302 с** (2026-06-17).

## 12. Замеры производительности и «наиболее полный тест»

### Ограничение прежних замеров К-6 / К-7

Замеры `07h`, `07h6`, `07f` и аналогичные прогоны **не включали** полноценное определение **плановых показателей** UtPl по всем пунктам ИП и всем контрольным датам цепи. В типичном prod-подобном наборе на 31.12 у большинства пунктов накопленный план в стеке **нулевой** (нет UtPl на активной ИП 11).

**Следствие:** оценки времени работы процедуры в целом (`spMstrg_2606`, `fn2`, полная цепь) **занижены** относительно сценария, когда организации заполняют помесячный UtPl.

### Наиболее полный тест (этап 18.7.4)

Единственный прогон, дающий репрезентативную картину **wall-clock** и соответствия требованиям **К-7** / интерактивности с учётом планов:

- цепь **5**, все **три** ИП (6, 8, 11) с тестовым UtPl в полном объёме (~1889 `ipgp`, ~90k строк `ipgUtPlPnLmMn`);
- матрица **17 дат** (`fnIpgChDats_2606`);
- проверки **К-12…К-17** + строгая логика **07n** / **07o** на полной цепи.

Ориентир времени: **30–90 мин** (зависит от dev-окружения); бюджет фиксируется отдельно при внедрении `run_plan_full_chain5.sh`.

До выполнения 18.7.4 заявления о соответствии полной процедуры требованиям по времени считаются **предварительными**.

## 16. Sparse-хранение UtPl и CHECK `lim > 0` (этап 18.8)

**Дата:** 2026-06-26 | **Решение:** `03-design-decisions.md` (Решение 15) | **Чат-план:** 18.8

### 16.1. Политика хранения

| Правило | Смысл |
|---------|--------|
| **Sparse** | В `ipgUtPlPnLmMn`, `ipgUtPlPnLmQu`, `ipgUtPlPnLmYe` хранятся **только** периоды с плановым освоением **> 0** |
| **Нет строки** | Отсутствие планового прироста в месяце/квартале/году — **не** нулевая строка, а **отсутствие записи** |
| **Расчёт** | `fnStCostUtPlPMn` / `fnStCostUtPlP`: нет строки → `0` (эквивалентно прежнему нулю в сетке) |
| **Сверка К-12** | `SUM(iuplpmLim)` по **существующим** строкам ≈ лимит `ipgPnLim`; полная сетка 12×4 **не требуется** |

**Запрещено:** вставка строк fixture/SQL с `iuplpmLim = 0` «для заполнения сетки» (подход FIXTURE_05/06 до 18.8.2).

### 16.2. Ограничения целостности (CHECK)

В схеме FishEye уже определены (сейчас **DISABLED** / NOT TRUSTED):

| Таблица | Ограничение | Целевое правило |
|---------|-------------|-----------------|
| `ipgUtPlPnLmMn` | `CK_ipgUtPlPnLmMn_gt0` *(было `<>0`)* | `iuplpmLim > 0` |
| `ipgUtPlPnLmQu` | `CK_ipgUtPlPnLmQu_gt0` | `iuplpqLim > 0` |
| `ipgUtPlPnLmYe` | `CK_ipgUtPlPnLmYe_gt0` | `iuplpyLim > 0` |

**Включение на dev:** этап **18.8.4** (`09c`), после очистки данных (**18.8.3**).  
**Включение на prod:** deploy-day **17.3.1** (`09a` → `09b` → `09c`).

### 16.3. Миграция данных (скрипты `09a`–`09c`, `MSSQL2012/`)

| Скрипт | Назначение | Dev | Prod |
|--------|------------|-----|------|

**Допущение prod:** с отпочкования dev новых ценных UtPl не появлялось → на prod ожидаются только **legacy**-нарушения (`lim < 0`, ~10 строк на момент аудита dev), **без** нулей fixture.

**Аудит dev (18.8.0):** 888 строк `lim = 0` (только FIXTURE_06); 9+1 строк `lim < 0` (legacy, округление mn=12 / Q4 @212).

### 16.4. Требования к fixture (после 18.8.2)
