## Ключевой принцип: от малого к большому

> **Никогда не запускать полную цепь (stIpg=NULL / 07f), пока не пройдены все блоки stIpg по отдельности.**

Производительность до оптимизации (Этап 7–8): `stIpg=46` (164 контракта) → **~42 мин**.  
Полная цепь → ~3 часа. Запуск без предварительной поблочной проверки = зависание и потеря времени.

### Этап 9б: сначала одна стройка, потом группа

После MSTVF v8 выяснилось: **К-6** упирается не в `fn2`, а в **сумму `fnMasteringCstAgPnSh_2606` по стройкам** (~4 с × 164).

**Стратегия:**

1. Зафиксировать **целевое время на стройку** (`T1`…`T★`, см. план чата §9б).
2. Идти **по одной стройке** (`07h2` + `run_07h2_contract.sh`), сравнивать с целью, дорабатывать `03b1`/`03c`/`03d`.
3. **Устойчивость:** 3 прогона подряд, разброс ≤20%.
4. После каждой доработки — `07h` по затронутому `stIpg` (корректность).
5. Переход к **групповому** `07h` только когда ≥90% строек группы укладываются в **T1** (промежуточно) или **T★** (К-6).

### 1. Перед запуском долгого теста — оценить время

Перед каждым запуском `fn2_2606` с незнакомым `stIpg`:
```sql
-- Сначала узнать количество контрактов:
SELECT COUNT(DISTINCT pp.ipgpCstAgPn)
FROM ags.ipgStPn sp
JOIN ags.ipgPn pp ON sp.ipgspPn = pp.ipgpKey
JOIN ags.ipgChRl cr ON cr.ipgcrIpg = pp.ipgpIpg AND cr.ipgcrChain = 5
WHERE sp.ipgspSt = @stIpg;
```
Умножить на эталонное время на контракт (до Этапа 8: ~15 сек/контракт).

### 2. Всегда использовать `RAISERROR WITH NOWAIT`

Все тестовые скрипты должны выводить прогресс через `RAISERROR(...) WITH NOWAIT`:
```sql
DECLARE @msg nvarchar(200);
SET @msg = N'[1/3] fn_2408 start: ' + CONVERT(nvarchar, SYSDATETIME(), 121);
RAISERROR(@msg, 0, 1) WITH NOWAIT;
```
Без этого SQL Server буферизует вывод и нет возможности отличить работу от зависания.

### 3. Перед `ALTER FUNCTION` — проверить активные сессии

```sql
SELECT session_id, status, command, wait_type, blocking_session_id,
       SUBSTRING(st.text, 1, 100) AS sql_text
FROM sys.dm_exec_requests r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) st
WHERE r.database_id = DB_ID('FishEye') AND session_id <> @@SPID;
```
Если есть долгоживущие сессии с нашими функциями — завершить (`KILL session_id`) перед `ALTER FUNCTION`.

### 4. Лимит ожидания: 3 минуты

Если тест работает дольше расчётного времени (×1.5) и нет вывода через RAISERROR — считать зависанием.  
Допустимо убить сессию и пересмотреть план.

### 5. `07f` — только финальный

`07f_COMPARE_PercentBrn_full_chain5.sql` — **никогда не запускать** до достижения К-7.  
Он вызывает `PercentBrn` дополнительно и работает в разы дольше `07h`.
